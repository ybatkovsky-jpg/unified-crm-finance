# Roadmap — Milestone v1.0 «ERP ПРО Мебель — доводка до спеки»

> Источник правды: `.planning/REQUIREMENTS.md` (51 REQ-ID) и `.gsd/integration/PRODUCT-SPEC.md`.
> Прежние заявки «M001–M008 готовы» — протухшие/фиктивные. Этот milestone доводит единый ERP до продуктового ТЗ.
> Стек: Next.js 16 + React 19 + Prisma 6 + PostgreSQL. Схема (58 моделей) и ~83 API-роута — недоделанный каркас: доводим/чиним + добавляем недостающее.

**Granularity:** Standard (10 фаз — обусловлено complex-проектом с 51 требованием и естественными границами модулей).
**Coverage:** 51/51 v1 требований замаплено.

## Phases

- [ ] **Phase 1: Доступ и авторизация (RBAC)** — экран входа, сессии, админка пользователей/ролей, middleware-изоляция.

## Phase Details

### Phase 1: Доступ и авторизация (RBAC)
**Goal**: Пользователь входит в систему по паре email/пароль, и его доступ к разделам/проектам ограничен ролью — без входа система закрыта.
**Depends on**: Nothing (хребет всего milestone).
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, AUTH-05
**Success Criteria** (what must be TRUE):
  1. Пользователь открывает любой URL системы без входа — его перенаправляет на `/login`; после ввода email+пароля он попадает в приложение, а после перезапуска браузера сессия сохраняется; кнопка «Выйти» очищает сессию.
  2. Директор в разделе «Настройки → Пользователи» создаёт нового пользователя, назначает ему роль, сбрасывает пароль и может заблокировать доступ — изменения применяются без перезапуска сервера.
  3. Пользователь с ролью «Менеджер-дизайнер» видит только свои проекты; попытка открыть чужой проект/раздел прямым URL возвращает 403/редирект (middleware проверяет матрицу видимости PRODUCT-SPEC п.1).
  4. Каждая роль (директор, менеджер-дизайнер, технолог, снабженец, монтажник, бухгалтер, кладовщик) видит ровно те пункты меню и разделы, что предписаны матрицей видимости.

### Phase 2: Стабилизация ядра
**Goal**: Базовые create-потоки приложения работают без 500, тесты перестают шуметь, а карточка сделки показывает корректный счётчик дней — фундамент доведён до состояния, на котором можно строить дальше.
**Depends on**: Phase 1
**Requirements**: CORE-01, CORE-02, CORE-03, CORE-04
**Success Criteria** (what must be TRUE):
  1. Пользователь успешно создаёт сделку, проект и инвойс через UI — ни один из этих потоков не возвращает 500 и не теряет данные.
  2. Запуск тестового набора (unit/integration) даёт «зелёный» прогон без удалённого/битого шума из ~298 прежних ошибок (мёртвые тесты удалены или переписаны).
  3. На карточке сделки отображается корректное «дни до конца проекта», рассчитанное из срока проекта.
  4. Проверка типов `tsc`/lint по прод-коду проходит без прежних ~40 schema/code drift ошибок (`assignedToId`, Category-импорт и т.п. устранены).

### Phase 3: Редизайн UI
**Goal**: После входа пользователь попадает в переработанный интерфейс с левым сайдбаром основных разделов, верхним поднавом подразделов и плавным, отзывчивым motion — работать приятно, в том числе с телефона.
**Depends on**: Phase 2
**Requirements**: UI-01, UI-02, UI-03, UI-04, UI-05
**Success Criteria** (what must be TRUE):
  1. Слева видны 7 основных разделов (CRM, Проекты, Закупки, Финансы, Учёт, Аналитика, Настройки); при выборе раздела сверху появляется таб-бар его подразделов по матрице PRODUCT-SPEC п.2.
  2. В шапке отображаются логотип «ПРО Мебель», рабочий глобальный поиск, текущий пользователь и кнопка «Выйти».
  3. Сразу после входа пользователь попадает на лендинг — канбан сделок CRM (а не на прежний top-bar экран).
  4. Карточки сделок, переходы между разделами и drag-and-drop движутся плавно через Framer Motion с мгновенным откликом; дизайн-токены (отступы, типографика, цвета статусов) зафиксированы в UI-SPEC.
  5. На мобильном экране монтажник/менеджер видит свои выезды, статусы и сдачу акта в удобном для телефона виде.

### Phase 4: CRM — сделки и КП
**Goal**: Менеджер-дизайнер ведёт сделку от первого обращения до договора: фиксирует источник, двигает по канбану, оформляет замер#1 и КП с версионированием, закрывает отказом с причиной, а при согласии получает договор + проект с общим номером `ПМ2026-0001` и начисленный бонус дизайнеру.
**Depends on**: Phase 3
**Requirements**: CRM-01, CRM-02, CRM-03, CRM-04, CRM-05, CRM-06, CRM-07, CRM-08
**Success Criteria** (what must be TRUE):
  1. При создании сделки пользователь выбирает источник из полного списка (2ГИС, сайт, интернет, Instagram, ВК, ТГ-группа, офис, сарафан, старая база, дизайнер).
  2. Пользователь перетаскивает сделку между стадиями воронки на канбан-доске; карточка показывает название, сумму, контакт, стадию проекта и «дни до конца проекта».
  3. Менеджер создаёт замер#1 как задачу-выезд (с переносом/пересозданием) и формирует КП из дизайн-проекта+материалов с версионированием (tier'ы по фурнитуре/материалам), PDF/распечаткой и сроком действия 30 дней.
  4. Сделку можно закрыть «отказ» только с обязательной причиной из справочника (дорого/конкурент/передумал/пропал/иное) — причина сохраняется для аналитики.
  5. Перевод сделки в «договор заключён» генерирует договор и создаёт проект со сквозным номером `ПМ2026-0001` (общим для проекта и договора); по проекту начисляется дизайнерский бонус 10% от суммы договора.

### Phase 5: Проект — спецификация и закупки
**Goal**: Технолог и снабженец проходят проект от замера#2 до закрытых по позициям закупок: детальный замер, единая спецификация с поставщиком по позиции, группировка закупок, email-запрос с автосверкой ответа и напоминаниями.
**Depends on**: Phase 4
**Requirements**: PROJ-01, PROJ-02, PROJ-03, PROJ-04, PROJ-05, PROJ-06, PROJ-07
**Success Criteria** (what must be TRUE):
  1. Проект проходит стадиями воронки Замер#2 → ТЗ/спецификация → Закупки → Производство → Монтаж → Акт → Закрытие (видимый статус и переходы).
  2. Технолог создаёт замер#2 как задачу-выезд (технолог+монтажник) и формирует единую спецификацию: позиция, материал, ед. изм., количество, поставщик в столбце, с возможностью ручного добавления позиций.
  3. Снабженец из спецификации получает закупки, сгруппированные по поставщику и с учётом наличия на складе; запрос поставщику проходит воронку Запрос отправлен → Получен счёт → Счёт оплачен → Получен товар.
  4. Запрос поставщику отправляется по email по шаблону, а ответ автоматически сверяется по наименованию/артикулу/кол-ву с уведомлением о расхождениях и отсутствующих позициях.
  5. Пока по спецификации остались позиции без закупок, проект визуально подсвечивается и напоминает (до полного покрытия).

### Phase 6: Производство, логистика, монтаж
**Goal**: Снабженец/менеджер ведут заказы на аутсорс-производства с учётом режима материала, видят доставку как расход проекта и управляют прогрессивным многозаходным монтажом, оформляя доп. работы формально.
**Depends on**: Phase 5
**Requirements**: PROJ-08, PROJ-09, PROJ-10, PROJ-11
**Success Criteria** (what must be TRUE):
  1. Производство-партнёр (контрагент с тегами-навыками: ДСП/МДФ, камень, стекло, бетон, покраска/плёнка, универсал) назначается на часть работ; заказ-наряд имеет режим материала «наш/партнёра» и проходит воронку Заказ передан → Материалы доставлены → В работе → Готово с контролем 60-дневного срока.
  2. Доставка поставщик→производство и производство→объект (свой транспорт) фиксируется, а её стоимость попадает в расходы проекта.
  3. Монтаж может стартовать после частичной доставки, фиксирует несколько заходов и статусы «приступили/закончили»; система требует получения финальных 30% перед началом монтажа.
  4. Доп. работы оформляются формально как доп. соглашение к договору или отдельный договор (только по инициативе клиента) и их стоимость попадает в оплату проекта.

### Phase 7: Акт, закрытие проекта, гарантия
**Goal**: Проект корректно закрывается: подписывается акт (монтажником для физлиц, менеджером для юрлиц), проверяются все условия закрытия и сохраняется срок гарантии.
**Depends on**: Phase 6
**Requirements**: PROJ-12, PROJ-13, PROJ-14
**Success Criteria** (what must be TRUE):
  1. Для физлица акт подписывается монтажником на объекте; для юрлица — менеджером-дизайнером (ЭДО/бумага); тип подписанта определяется по типу контрагента.
  2. Проект нельзя закрыть, пока не выполнены одновременно: акт подписан, все деньги клиента получены, все счета поставщикам/производствам оплачены, бонус дизайнеру выплачен.
  3. По проекту хранится срок гарантии 2 года (+ фурнитура по гарантии производителя), и он виден на карточке проекта.

### Phase 8: Финансы
**Goal**: Бухгалтер ведёт платежи проекта (70% + 30% + наличные), загружает банк-выписку файлом, видит маржу проекта с алертами, долги (дебиторка/кредиторка) и выплату дизайнеру.
**Depends on**: Phase 7
**Requirements**: FIN-01, FIN-02, FIN-03, FIN-04, FIN-05, FIN-06
**Success Criteria** (what must be TRUE):
  1. Платежи клиента (70% предоплата + 30% перед монтажом) и наличные привязываются к проекту; способ оплаты «наличные» вносится вручную.
  2. Импорт банк-выписки файлом 1С/TXT (Озон, Тинькофф) парсится и сопоставляет платежи проектам/счетам по контрагенту/сумме/назначению с подтверждением пользователем.
  3. Маржа проекта считается как доходы − расходы (материалы, производства, доставка, бонус, доп. работы); проекты ниже порога подсвечиваются алертом.
  4. Видны долги: дебиторка (клиенты) и кредиторка (поставщики/производства/дизайнер) с суммой и датой просрочки.
  5. Бонус дизайнеру выплачивается разово по проекту после всех денег клиента; виден накопленный долг дизайнера по нескольким проектам.

### Phase 9: Управленческий учёт
**Goal**: Директор видит здоровье организации за период: 12 постоянных статей расходов, P&L, план/факт по статьям и ДДС план/факт.
**Depends on**: Phase 8
**Requirements**: ACCT-01, ACCT-02, ACCT-03, ACCT-04
**Success Criteria** (what must be TRUE):
  1. В справочнике ведутся 12 постоянных статей расходов (зерно PRODUCT-SPEC п.6: аренда, бухгалтери­я, зарплаты, реклама и т.д.).
  2. Отчёт P&L за период показывает доходы (по проектам + иное) минус расходы (постоянные + проектные) с итоговой прибылью/убытком.
  3. План/факт ведётся по статьям и периодам — видно отклонение факта от плана.
  4. Отчёт ДДС (движение денежных средств) представлен в режиме план/факт (когда деньги придут/уйдут), отдельно от P&L.

### Phase 10: Задачи, уведомления, аналитика
**Goal**: Команда работает с задачами-выездами, получает уведомления по ключевым событиям в нужных каналах, а директор видит аналитику воронки, маржи и команды.
**Depends on**: Phase 9
**Requirements**: PLAT-01, PLAT-02, PLAT-03, PLAT-04, PLAT-05
**Success Criteria** (what must be TRUE):
  1. Задачи (выезды замер#1/#2/монтаж и общие) имеют исполнителя, дедлайн, статус; их можно перенести и пересоздать при неудаче.
  2. По ключевым событиям (лид, смена стадии, оплата, дедлайн 60 дней, ответ поставщика/расхождение, просрочка задачи) уходят уведомления в Telegram/in-app/email.
  3. Аналитика воронки продаж показывает конверсию по стадиям и разбивку причин отказов.
  4. Аналитика маржи показывает текущую/закрытую маржу проектов и алерты по низкомаржинальным.
  5. Аналитика команды показывает нагрузку и результативность участников (менеджеры, технолог, монтажники, производства).
