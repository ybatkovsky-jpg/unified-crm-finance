# M006: Финансы

**Gathered:** 2026-06-24
**Status:** Ready for planning

## Project Description

Модуль **Финансы** для unified-crm-finance — учёт денежных операций, аналитика и планирование для малого бизнеса с проектной-centric моделью. Включает:

- **Dashboard** — 3 слоя: оперативка (сальдо, платежи 7 дней, burn rate), текущий месяц (доходы/расходы vs бюджет), аналитика (P&L, Cash Flow, EBITDA по клику)
- **Транзакции** — единая таблица всех доходов/расходов с фильтрами по проекту/категории/дате
- **Тип оплаты** — безнал (99%, авто из выписок 1C), наличные (1%, ручной ввод)
- **Проектная фин. карточка** — мини-дашборд (доходы, расходы, маржа, рентабельность) + таблица транзакций проекта
- **Импорт 1C ClientBank** — реальный парсер TXT с дедупликацией и автопривязкой к Invoice → Project
- **Рекуррентные расходы** — авто-создание по расписанию (арена — фиксированная, остальное ручное)
- **Бюджеты** — план на месяц/квартал/год для контроля без лимитов
- **Закрытие периодов** — owner может редактировать всё, остальные — только открытые периоды

## Why This Milestone

Единая система для учёта финансов компании вместо разрозненных таблиц и 1C. Видеть, куда уходят деньги, планировать бюджеты, отслеживать исполнение.

## User-Visible Outcome

### When this milestone is complete, the user can:

- Видеть Dashboard с оперативкой (сальдо, платежи 7 дней, burn rate), бюджетом текущего месяца и кнопкой аналитики (P&L, Cash Flow, EBITDA)
- Создавать/просматривать/редактировать транзакции через UI по маршруту `/finance/*`
- Импортировать банковские выписки из 1C ClientBank (TXT формат, реальный парсер с дедупликацией)
- Оплачивать счета из модуля Закупки или из модуля Финансы с созданием связанной транзакции
- Планировать бюджеты на месяц/квартал/год и видеть прогресс исполнения (предупреждение при превышении, без блокировок)
- Просматривать отчёты (движение денег, P&L, Cash Flow) в виде таблиц с фильтрами по дате/категории/контрагенту/проекту
- Открывать карточку проекта и видеть мини-дашборд (доходы, расходы, маржа, рентабельность) + таблицу транзакций проекта
- Создавать рекуррентные расходы (арена — фиксированная, прочее — ручной ввод) с авто-созданием по расписанию
- Редактировать любые транзакции (owner) или только открытые периоды (остальные)

## Architectural Decisions

### Валюта

**Decision:** Только рубли для MVP

**Rationale:** Упрощает схему и UI. Многовалютность требует ExchangeRate, исторических курсов, пересчёта — можно добавить позже.

**Alternatives Considered:**
- Многовалютность с историческими курсами — избыточно для MVP
- Хранение курсов с пересчётом в реальном времени — усложняет архитектуру

### Бюджетный контроль

**Decision:** Предупреждение при превышении, без блокировок

**Rationale:** Контроль без жёстких ограничений. Пользователь видит превышение, но может провести операцию.

**Alternatives Considered:**
- Жёсткий блок при превышении — слишком ограничительно для реального бизнеса
- Только визуальный индикатор — недостаточно для контроля

### Статусы транзакций

**Decision:** Автоматический переход + ручной режим

**Rationale:** pending → classified (после назначения категории) → confirmed (после подтверждения). Пользователь может вручную изменить статус при необходимости.

**Alternatives Considered:**
- Только автоматический — недостаточно гибкости
- Только ручной — лишние действия пользователя

### Дедупликация транзакций

**Decision:** Автоматическое определение по сумме + дате + контрагенту

**Rationale:** Повторный импорт того же файла не должен создавать дубликаты. Хеш-ключ по трём полям надёжен для большинства случаев.

**Alternatives Considered:**
- Ручное удаление дубликатов — неудобно для пользователя
- Игнорировать проблему — засорение базы

### Интеграция с Invoice

**Decision:** Оплата возможна из обоих модулей (Закупки и Финансы)

**Rationale:** Гибкость UX. В Закупке — на странице счёта. В Финансах — на странице платежей.

**Alternatives Considered:**
- Только из Закупки — неудобно, если пользователь работает в Финансах
- Только из Финансов — разрывает workflow в Закупке

### Тип оплаты (нал/безнал)

**Decision:** Безнал определяется источником (выписка 1C), наличные — ручной ввод

**Rationale:** 99% транзакций — безнал из банковских выписок. Наличные — редкий кейс для мелких расходов. Тип определяется автоматически при импорте или вручную при создании.

**Alternatives Considered:**
- Отдельная колонка в UI — избыточно, так как 99% = безнал
- Игнорировать тип оплаты — теряется видимость наловых операций

### Рекуррентные расходы

**Decision:** Авто-создание по расписанию для фиксированных (арена), ручной ввод для переменных

**Rationale:** Аренда — фиксированная сумма ежемесячно. ЗП, бухгалтерия — зависят от объёма, требуют ручного ввода. Крон-задача проверяет расписание и создаёт Transaction.

**Alternatives Considered:**
- Всё рекуррентное — не подходит для переменных расходов
- Всё ручное — лишние действия для фиксированных платежей

### Закрытие периодов

**Decision:** Owner редактирует всё, остальные — только открытые периоды

**Rationale:** Owner должен иметь возможность корректировать любые периоды. Сотрудники работают только с текущим открытым периодом для целостности данных.

**Alternatives Considered:**
- Все редактируют всё — риск ошибок в закрытых периодах
- Никто не редактирует — не гибко для реального бизнеса

## Error Handling Strategy

- Ошибки парсера 1C ClientBank — показать пользователю конкретную строку/причину ошибки, не прерывать импорт полностью
- Ошибка API при создании транзакции — показать toast с сообщением, сохранить форму
- Превышение бюджета — показать warning banner, разрешить действие
- Дубликат транзакции — автоматически пропустить, добавить в лог "пропущено N дубликатов"

## Risks and Unknowns

- **Формат 1C ClientBank** — формат TXT файла не стандартизирован; нужен пример реального файла для тестирования парсера
- **Крон-задача для рекуррентных расходов** — не определён механизм запуска (node-cron, RabbitMQ scheduled task, Vercel Cron)
- **Закрытие периодов** — не определён механизм (флаг isClosed на Transaction, отдельная сущность Period, или просто проверка дат)

## Integration Points

- **Invoice (модуль Закупка M005)** — оплата счёта создаёт Transaction + CashFlowPayment, обновляет Invoice.paidAt, автопривязка к Project через Invoice
- **Project (модуль Проекты M003)** — Transaction.projectId связывает транзакцию с проектом; на карточке проекта показывается мини-дашборд (доходы, расходы, маржа, рентабельность)
- **Counterparty (модуль Закупка M005)** — Transaction.counterpartyId для фильтрации по контрагенту; также связан с Contact из M002
- **Contact (модуль CRM M002)** — источник Counterparty, используется для идентификации плательщика/получателя
