# S02: Contact List UI ‚Äî Research

**Date:** 2026-06-21

## Summary

S02 builds the `/crm/contacts` page ‚Äî a table of contacts with filters by type (person/company) and lead source.
The Contact CRUD API from S01 is complete and ready to consume via `ContactApiClient`.
However, the apps/web project has **no frontend shell at all**: no `layout.tsx`, no `page.tsx`, no components directory, and critically **shadcn/ui is not installed** (no `components.json`, no Tailwind CSS, no Radix primitives in `package.json`).
The first task for S02 must bootstrap the entire UI stack: install Tailwind CSS + shadcn/ui + lucide-react, then create the app layout shell and the CRM contacts page.

## Recommendation

**Bootstrap the frontend stack first, then build the contacts page.** Specifically:
1. Install and configure Tailwind CSS 4 (Next.js 16 compatible), shadcn/ui, lucide-react
2. Create a minimal root `layout.tsx` with HTML shell
3. Build `/crm/contacts/page.tsx` as a client component using `ContactApiClient` for data fetching, rendering a filterable table with shadcn/ui Table, Select (for type/source filters), and Button components

Client-side data fetching via `ContactApiClient` is the right approach because:
- The S01 API client is already tested and typed ‚Äî zero duplication
- Client components support interactive filtering without full page reloads
- The API routes are public (no auth middleware yet), so no server-side session requirements block client fetching

### Build Order

2. **Root layout + utils** ‚Äî Create `src/app/layout.tsx`, `src/lib/utils.ts` (cn helper). Provides the shell every page renders into.
3. **Contact list page** ‚Äî Create `src/app/crm/contacts/page.tsx` with ContactApiClient data fetching, shadcn/ui Table, type/status filter Selects, loading/empty/error states. Links each row to `/crm/contacts/[id]` (S04's page).

### Verification Approach

- **Test**: Optionally add a smoke test that mocks `contactsApi.getContacts()` and renders the page

## Constraints

- **shadcn/ui is not yet initialized** ‚Äî the full UI stack (Tailwind, PostCSS, Radix primitives) must be bootstrapped from scratch. This is the biggest risk and should be T01.
- **No auth middleware** ‚Äî API routes are public, so client-side fetch works without auth tokens in MVP. Will need to revisit when M001 auth is wired up.
- **No pagination in API** ‚Äî GET /api/contacts returns all records. For MVP this is acceptable; pagination should be added later (separate task or follow-up).
- **LeadSource filter requires source data** ‚Äî The `LeadSource` table exists but may be empty. Either seed default sources or make the filter optional until data exists.
- **Next.js 16 with Turbopack** ‚Äî uses new configuration patterns. The S01 team installed Next.js locally in apps/web. Same pattern applies here.

## Common Pitfalls

- **Tailwind CSS v4 configuration** ‚Äî Next.js 16 may use Tailwind v4 which has a different config format (CSS-based instead of `tailwind.config.js`). Check the installed version and use the correct setup.
- **shadcn/ui init in monorepo** ‚Äî The `npx shadcn@latest init` command needs to run in `apps/web`, not the monorepo root. Path aliases must match `@/*` ‚Üí `./src/*` from the existing tsconfig.
- **Client vs Server component** ‚Äî The contacts page needs interactive filters, so it must be a client component (`'use client'`). But the root `layout.tsx` should be a server component.
- **contactId links** ‚Äî S02 links rows to `/crm/contacts/[id]` but that page is built in S04. Make links forward-compatible (they'll 404 until S04 is done).

## Open Risks

- shadcn/ui init may fail or produce incompatible output with Next.js 16 Turbopack ‚Äî need to verify immediately after bootstrap
- The `apps/web/src/lib/db.ts` file (old Prisma singleton) conflicts with `apps/web/src/lib/db/prisma.ts` (S01's singleton) ‚Äî not directly blocking S02 but should be cleaned up
- No seed data for LeadSource ‚Äî contact source filter will show empty dropdown until data exists
