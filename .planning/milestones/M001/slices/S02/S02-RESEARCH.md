# S02 — Research

**Date:** 2026-06-20

## Summary

S02 implements the complete Prisma data model with 42 entities across 7 bounded contexts: Identity (User, Role, Permission, UserRole), CRM (Company, Contact, Lead), Sales (Deal, DealStage, DealActivity, ActivityAttachment), Contracts (Contract, ContractTemplate, ContractVersion, ContractSignatory), Projects (Project, ProjectMilestone, ProjectTask), Procurement (ProcurementRequest, Supplier, PurchaseOrder, Invoice, Payment, Budget, ApprovalRequest, Document), Finance (Account, Transaction, Reconciliation, TaxRule, FinancialReport), and Shared (FileEntity, Comment, Tag, Category, Notification, AuditLog).

The primary recommendation is a **phased implementation approach**: start with core Identity models (User, Role, Permission, UserRole) to validate Prisma + PostgreSQL integration before adding all 42 entities. This mitigates risk by proving the toolchain works before complexity increases. PostgreSQL 16's native `gen_random_uuid()` support simplifies UUID primary keys without requiring the `pgcrypto` extension.

The Docker Compose DATABASE_URL must use `postgres` hostname (not `localhost`) for container networking: `postgresql://crm:crm_password@postgres:5432/crm?schema=public`. Use `npx prisma migrate dev` for development and `npx prisma migrate deploy` for production. Verify with `npx prisma studio` for visual schema inspection.

## Recommendation

Implement Prisma schema in **7-8 phased migrations by bounded context** rather than one monolithic migration. Each migration adds 3-6 related entities, making debugging and rollback easier. Start with Identity (validates auth foundation), then Shared (cross-cutting entities used by all modules), then business domains in dependency order. This approach isolates problems: if Deal entities fail, Contracts and Projects remain unaffected.

Use **Prisma as source of truth** for the hybrid architecture (D006). Python's SQLAlchemy will read from the same database using read-mostly views. ADR-002 must document this relationship explicitly: Prisma owns schema evolution, SQLAlchemy mirrors for worker access.

## Implementation Landscape

### Key Files

- `apps/web/prisma/schema.prisma` — Complete 42-entity Prisma schema with relations, indexes, and constraints
- `apps/web/prisma/migrations/*` — SQL migrations generated by `npx prisma migrate dev`, one per bounded context
- `apps/web/prisma/migrations/M001_*_down.sql` — Rollback migrations (Prisma only supports single-step rollback)
- `.gsd/adr/002-data-model.md` — ADR documenting Prisma + SQLAlchemy consistency strategy
- `apps/web/.env.example` — Add Prisma-specific env vars (DATABASE_URL, NEXTAUTH_SECRET)

### Build Order

1. **Add Prisma to apps/web/package.json** — `@prisma/client@^6.0.0` and `prisma@^6.0.0` as dependencies (not devDependency — needed at runtime)
2. **Create apps/web/prisma/schema.prisma with Identity entities** — User, Role, Permission, UserRole with UUID primary keys and indexes
3. **Run `npx prisma migrate dev --name 001_identity`** — Generates migration and creates `dev` database in Docker
4. **Verify with `npx prisma studio`** — Visual inspection of User, Role, Permission tables
5. **Add Shared entities to schema.prisma** — FileEntity, Comment, Tag, Category, Notification, AuditLog
6. **Run `npx prisma migrate dev --name 002_shared`** — Second migration
7. **Add CRM entities to schema.prisma** — Company, Contact, Lead
8. **Continue for Sales, Contracts, Projects, Procurement, Finance** — One migration per bounded context
9. **Write ADR-002** — Document Prisma + SQLAlchemy consistency approach

### Verification Approach

- `npx prisma migrate dev` completes without errors for all 7-8 migrations
- `npx prisma studio` opens and displays all 42 entities with correct relations
- `npx prisma generate` produces TypeScript types matching schema (check `node_modules/.prisma/client/index.d.ts`)
- Docker Compose `postgres` container logs show successful migrations: `database "crm" is up to date at migration X`
- Query test: `npx prisma db execute --sql "SELECT count(*) FROM \"User\""` returns `0`

## Don't Hand-Roll

| Problem | Existing Solution | Why Use It |
|---------|------------------|------------|
| UUID primary keys | PostgreSQL `gen_random_uuid()` default | Native, no extension needed, faster than application-generated UUIDs |
| Relation management | Prisma's `@relation` decorator | Handles foreign keys, cascading deletes, and TypeScript types automatically |
| Migration rollback | `npx prisma migrate resolve --rolled-back <migration>` | Prisma only supports single-step rollback; this marks migration as applied without running it |
| Database access | Prisma Client (`@prisma/client`) | Type-safe queries, generated TypeScript types, connection pooling built-in |

## Constraints

- **PostgreSQL 16 required** — `gen_random_uuid()` is native in PG 16+; older versions need `pgcrypto` extension
- **Docker networking** — DATABASE_URL must use `postgres:5432` hostname, not `localhost:5432`
- **Migration order matters** — Entities with foreign keys must be migrated after their targets (Identity before everything, Shared before domain entities)
- **Prisma 6 only** — Next.js 16 requires Prisma 6+; Prisma 5 has incompatible edge runtime handling

## Common Pitfalls

- **Using `localhost` in DATABASE_URL** — Works locally but fails in Docker Compose. Always use `postgres` hostname.
- **Forgetting `prisma generate` after migrations** — TypeScript types won't update. Post-migration, regenerate client.
- **Circular relations without explicit `fields`** — Prisma can't infer relation fields in cycles (Deal ↔ Contract ↔ Project). Always specify both sides.
- **Missing indexes on foreign keys** — Prisma doesn't auto-index relations. Add `@@index([column])` for query performance.
- **Running migrations out of order** — If migration 003 depends on 002 tables, running 003 first fails. Follow bounded context order.

## Open Risks

- **Prisma + SQLAlchemy consistency** — Two ORMs on one database could diverge. Mitigated by making Prisma source of truth and ADR-002 documentation.
- **Migration rollback complexity** — Prisma only supports single migration rollback. Phased approach limits blast radius of rollback failures.
- **Circular dependencies in entities** — Deal ↔ Contract ↔ Project form cycles. Careful relation design with explicit `fields` required.
- **Large schema type generation time** — 42 entities with many relations may slow `prisma generate`. Phased approach adds incremental generation time.
