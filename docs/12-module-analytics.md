# 12. Модуль Аналитика и дашборды

## 12.1. Назначение

Модуль Аналитика — сквозная аналитика по всем модулям: воронка продаж, маржинальность проектов, оборачиваемость склада, эффективность менеджеров, ROI по источникам лидов. Это отдельный модуль, потому что многие срезы требуют агрегации данных из разных контекстов (CRM + Sales + Procurement + Finance) и тяжёлых запросов, которые не должны влиять на транзакционную нагрузку.

## 12.3. User Stories

- **US-ANL-01**: Как владелец, я хочу видеть на главной странице все ключевые метрики бизнеса, чтобы не открывать разные модули.
- **US-ANL-02**: Как владелец, я хочу видеть воронку продаж визуально, чтобы понимать, на каких стадиях теряются сделки.
- **US-ANL-03**: Как владелец, я хочу сравнить эффективность менеджеров, чтобы понимать, кого нужно обучить.
- **US-ANL-04**: Как владелец, я хочу получить ежемесячный отчёт по email, чтобы не заходить в систему.
- **US-ANL-05**: Как менеджер, я хочу видеть динамику маржинальности своих проектов, чтобы понимать, какие требуют внимания.

## 12.4. UI/UX

### 12.4.1. Главная страница

- Селектор периода (top-right).
- Сетка виджетов (drag-and-drop для кастомизации):
  - Карточка: Выручка, Расходы, Маржа, Активные сделки.
  - График: Выручка/Расходы по месяцам.

- Визуальная воронка (горизонтальная).
- Таблица под воронкой: Стадия | Кол-во | Сумма | Конверсия в следующую | Средний чек.
- Фильтры: период, воронка, менеджер.

### 12.4.3. Страница «Команда»

- Таблица менеджеров с метриками.
- Bar chart: конверсия по менеджерам.

## 12.5. Технические аспекты

### 12.5.1. Материализованные представления

Для тяжёлых агрегатов используются PostgreSQL materialized views, обновляемые раз в час (через Celery beat):

```sql
-- mv_deal_funnel
CREATE MATERIALIZED VIEW mv_deal_funnel AS
SELECT
  pipeline_id,
  stage_id,
  COUNT(*) AS deal_count,
  SUM(amount) AS total_amount,
  AVG(amount) AS avg_amount
FROM deals
WHERE deleted_at IS NULL
GROUP BY pipeline_id, stage_id;

-- mv_project_pnl
CREATE MATERIALIZED VIEW mv_project_pnl AS
SELECT
  p.id AS project_id,
  p.contract_amount,
  COALESCE(SUM(CASE WHEN t.type='income' THEN t.amount ELSE 0 END), 0) AS total_income,
  COALESCE(SUM(CASE WHEN t.type='expense' THEN t.amount ELSE 0 END), 0) AS total_expense,
  COALESCE(SUM(CASE WHEN t.type='income' THEN t.amount ELSE -t.amount END), 0) AS profit,
  CASE WHEN p.contract_amount > 0
       THEN COALESCE(SUM(CASE WHEN t.type='income' THEN t.amount ELSE -t.amount END), 0) / p.contract_amount
       ELSE NULL END AS margin
FROM projects p
LEFT JOIN transactions t ON t.project_id = p.id AND t.deleted_at IS NULL
WHERE p.deleted_at IS NULL
GROUP BY p.id, p.contract_amount;

-- mv_cashflow_calendar
CREATE MATERIALIZED VIEW mv_cashflow_calendar AS
SELECT
  date_trunc('day', date) AS day,
  project_id,
  SUM(CASE WHEN type='inflow' OR (type='planned_inflow' AND status='confirmed') THEN amount ELSE 0 END) AS inflow,
  SUM(CASE WHEN type='outflow' OR (type='planned_outflow' AND status='confirmed') THEN amount ELSE 0 END) AS outflow
FROM cash_flow_payments
GROUP BY date_trunc('day', date), project_id;
```

### 12.5.2. Кеширование

- Результаты запросов кешируются на 5 минут в Redis (если включён) или in-memory.
- Кеш инвалидируется при изменениях в соответствующих таблицах (через Prisma middleware).

### 12.5.3. Производительность

- Все аналитические запросы используют индексы по дате и проекту.
- Сложные срезы (например, ROI по источникам) выполняются асинхронно с прогрессом.
- Для больших объёмов данных (>1М транзакций) — партиционирование по дате.

## 12.8. Метрики модуля (мета-метрики)

- Дашборд открывается <2 секунд.
