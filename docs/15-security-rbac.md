# 15. Безопасность и RBAC

## 15.1. Угрозная модель

Система обрабатывает:
- Персональные данные клиентов (ФИО, телефоны, email, адреса) — 152-ФЗ.
- Коммерческую тайну (суммы договоров, маржинальность, условия с поставщиками).
- Финансовые данные (банковские выписки, платежи).
- Учётные данные пользователей (хеши паролей).
- API-ключи внешних сервисов (LLM, SMTP, Telegram).

Угрозы:
1. Утечка БД через SQL-инъекцию.
2. Утечка токенов и ключей через код или логи.
3. Несанкционированный доступ (украденный пароль, отсутствие RBAC).
4. CSRF-атаки на формы.
5. XSS через пользовательский ввод (имена, заметки).
6. Brute-force паролей.
7. Превышение лимитов (rate limit) на ресурсоёмких эндпоинтах (LLM, экспорты).
8. Утечка данных при увольнении сотрудника (доступ остаётся).
9. Insider-атака (менеджер скачивает всю базу контактов перед уходом).

## 15.2. Аутентификация

### 15.2.1. Пароли

- Хеширование — bcryptjs с cost factor 12 (или argon2id, если решим мигрировать).
- Минимальная длина — 10 символов.
- Требования к сложности: минимум 1 буква, 1 цифра, 1 спецсимвол.
- Запрет на повторное использование последних 5 паролей.
- Срок действия — без принудительного истечения (рекомендация NIST).
- Восстановление пароля — через email-ссылку с токеном (TTL 1 час, одноразовый).

### 15.2.2. Сессии

- Access-токен (JWT) — TTL 15 минут.
- Refresh-токен — TTL 7 дней, хранится в БД (`RefreshToken`), можно отозвать.
- При logout — refresh-токен ревёкся.
- При обнаружении подозрительной активности (несколько IP одновременно) — принудительный logout.

### 15.2.3. Двухфакторная аутентификация (2FA)

- Опционально, для ролей owner и accountant — обязательно.
- Реализация — TOTP (Google Authenticator / Authy).
- Backup-коды — 10 штук при включении 2FA.

### 15.2.4. API-ключи

- Для интеграций — отдельные API-ключи (`X-API-Key` заголовок).
- Ключи хранятся в БД в хешированном виде (как пароли).
- При создании ключа — показывается один раз, далее только хеш.
- Можно ограничить права ключа (scope): только чтение, только определённые модули.

## 15.3. Авторизация (RBAC)

### 15.3.1. Роли

| Код роли | Название | Описание |
|----------|----------|----------|
| `owner` | Владелец | Полный доступ ко всему, одобряет заявки, видит финансы |
| `sales` | Менеджер по продажам | Работает с контактами и сделками, видит свои данные |
| `manager` | Менеджер проектов | Ведёт проекты, закупки, видит свои и чужие данные по проектам |
| `accountant` | Бухгалтер | Импорт выписок, классификация, оплата счетов |
| `storekeeper` | Кладовщик | Склад, приёмка, списание |

### 15.3.2. Матрица прав

Полная матрица прав по модулям — в каждом разделе модуля (см. docs/06–13). Здесь — общие принципы.

| Ресурс | owner | sales | manager | accountant | storekeeper |
|--------|-------|-------|---------|------------|-------------|
| Пользователи (CRUD) | да | нет | нет | нет | нет |
| Настройки системы | да | нет | нет | нет | нет |
| Контакты | все | свои+неприкр. | все | все | нет |
| Сделки | все | свои | все | все | нет |
| Договоры | все | свои | все | все | нет |
| Проекты | все | свои | все | все | свои |
| Спецификации | да | нет | да | да | да (чтение) |
| Запросы поставщикам | да | нет | да | нет | нет |
| Счета | да | нет | да | да | нет |
| Согласования | да | нет | нет | нет | нет |
| Склад | да | нет | да | нет | да |
| Транзакции | все | нет | свои | все | нет |
| Бюджеты | да | нет | да | да | нет |
| Отчёты | все | свои | все | все | нет |
| Уведомления | свои | свои | свои | свои | свои |

### 15.3.3. Реализация

- На уровне Next.js middleware: проверка JWT, извлечение userId, проверка ролей.
- На уровне route handlers: helper `requireRole(user, 'owner')`, выбрасывает 403 при несоответствии.
- На уровне Prisma-запросов: фильтр по `ownerId = currentUser.id` для sales, без фильтра для owner.
- На уровне отдельных полей: например, `salary` сотрудников видит только owner (через DTO-трансформацию).

### 15.3.4. Audit

Все мутации (create/update/delete) пишутся в `AuditLog`:
- `userId` — кто.
- `entityType` + `entityId` — над чем.
- `action` — что сделал.
- `changes` — `{ before: {...}, after: {...} }` (только изменённые поля).
- `metadata` — IP, User-Agent, source (web/api/internal).

Чтение аудита — только owner. Возможно выгружать в CSV/Excel.

## 15.4. Защита от атак

### 15.4.1. SQL-инъекции

- Все запросы к БД — только через Prisma (параметризованные).
- Сырой SQL — только в миграциях, без пользовательского ввода.
- На уровне Prisma middleware — логирование всех сырых запросов (для аудита).

### 15.4.2. XSS

- React по умолчанию экранирует вывод.
- `dangerouslySetInnerHTML` — только для Markdown-рендера (через `react-markdown` с sanitization).
- Sanitize HTML-ввод через `DOMPurify` (для notes, descriptions).
- Content-Security-Policy header: только self + доверенные домена (S3, LLM API).

### 15.4.3. CSRF

- Double-submit cookie: `csrf_token` в cookie и в header `X-CSRF-Token` для всех POST/PUT/DELETE.
- SameSite=Lax на всех cookies.
- На API-key эндпоинты CSRF не требуется (только для cookie-based).

### 15.4.4. Brute-force

- Rate limit на `/auth/login`: 5 попыток в минуту на IP, 10 в час на email.
- После 5 неудач — Captcha (hCaptcha или Cloudflare Turnstile).
- После 10 неудач — блокировка аккаунта на 30 минут + уведомление owner.

### 15.4.5. Rate limiting (общий)

- 100 запросов в минуту на пользователя (JWT).
- 200 запросов в минуту на IP (без JWT).
- На экспортные эндпоинты — 5 запросов в минуту.
- На AI-эндпоинты (загрузка BOM, сверка счетов) — 10 запросов в минуту.

### 15.4.6. Path traversal

- Все пути к файлам — генерируются сервером, не от пользователя.
- Загруженные файлы сохраняются в S3 под сгенерированными ключами (UUID).
- Имя файла пользователя — только для отображения, sanitize через `path.basename()`.

### 15.4.7. File upload

- Ограничение размера: 10 МБ для Excel, 50 МБ для PDF, 100 МБ всего на запрос.
- Проверка MIME-типа (не только расширения).
- Антивирусная проверка — ClamAV (опционально, на следующей фазе).

## 15.5. Секреты и конфигурация

### 15.5.1. Хранение

- Все секреты (пароли к БД, API-ключи, JWT-секрет) — в переменных окружения.
- В коде — никаких захардкоженных секретов.
- `.env` файл — в `.gitignore`, в репозитории только `.env.example`.
- Для production — секреты в Docker secrets / Kubernetes secrets / Vault.

### 15.5.2. Ротация

- JWT-секрет — ротация раз в 90 дней (с grace period 7 дней для старых токенов).
- API-ключи внешних сервисов — ротация по графику каждого провайдера.
- Пароль БД — ротация раз в 6 месяцев.

### 15.5.3. Логирование

- В логи НИКОГДА не пишутся: пароли, токены, API-ключи, номера карт, CVV.
- Если в ошибке есть sensitive-данные — маскируются (например, `password: "***"`).
- Структурированные логи (JSON) с уровнями: debug, info, warn, error.
- В production — уровень info, debug только по запросу.

## 15.6. Сетевая безопасность

### 15.6.1. TLS

- Все соединения — только HTTPS (Caddy с auto-HTTPS).
- Минимум TLS 1.2, рекомендуется 1.3.
- HSTS header: `max-age=31536000; includeSubDomains; preload`.

### 15.6.2. Заголовки безопасности

```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https://api.openai.com https://api.anthropic.com; font-src 'self' data:;
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=(), camera=()
```

### 15.6.3. Изоляция сервисов

- Python-воркер и Next.js — в разных Docker-контейнерах.
- Внутренние webhook'и между ними — через `X-Internal-Secret` (env var).
- Доступ к БД — оба сервиса, но с разными пользователями:
  - `web_user` — полный доступ (CRUD).
  - `worker_user` — только на нужные таблицы (read + write в `invoices`, `bom_items`, `audit_logs`).

### 15.6.4. CORS

- `Access-Control-Allow-Origin` — только домен приложения.
- Для API-key запросов — можно разрешить дополнительные домены (настраивается).

## 15.7. Резервное копирование

### 15.7.1. База данных

- Ежедневный full dump через `pg_dump` — в 03:00.
- WAL archiving — для point-in-time recovery.
- Бэкапы хранятся в S3 (отдельный bucket), retention 30 дней.
- Еженедельная проверка restore на staging.

### 15.7.2. Файлы (S3)

- Versioning включён.
- Cross-region replication (опционально).
- Lifecycle: transition to Glacier после 90 дней неактивности, delete после 1 года (для архивных).

### 15.7.3. RTO / RPO

- RTO (Recovery Time Objective) — 4 часа.
- RPO (Recovery Point Objective) — 24 часа (дневной бэкап) + WAL archiving (до 1 часа потерь).

## 15.8. Соответствие 152-ФЗ

- Персональные данные клиентов (ФИО, телефоны, email, адреса) — хранятся в РФ-юрисдикции (сервер в РФ).
- Согласие на обработку фиксируется:
  - В `Contact.attributes.consentAt`, `consentText` — при создании контакта менеджером (читается с сайта/формы).
  - При создании через сайт — чекбокс «Согласен на обработку» обязателен.
- Право на удаление — по запросу клиента: soft-delete контакта, удаление из бэкапов по графику (30 дней).
- Журнал доступа к ПД — отдельная таблица `AuditLog` с `entityType=contact`.

## 15.9. Безопасность AI-агента

### 15.9.1. Human-in-the-loop

- AI-агент НИКОГДА не отправляет email напрямую — все отправки через `ApprovalRequest` (если не отключено в настройках).
- AI-агент НЕ списывает деньги, не создаёт платёжные поручения.
- Все решения AI (сверка счёта, парсинг BOM) — помечаются как «черновик», требуют подтверждения менеджером.

### 15.9.2. Промпт-инъекции

- Пользовательский ввод (имена, описания) — не передаётся в LLM-промпт напрямую без sanitization.
- LLM-вывод — валидируется через Pydantic-схему, невалидный — отбрасывается, задача в DLQ.

### 15.9.3. Логирование LLM-вызовов

- Все вызовы LLM — в таблицу `LlmLogs`: промпт, ответ, токены, стоимость, провайдер, дата.
- Это позволяет отслеживать расходы и расследовать ошибки.

## 15.10. Безопасность интеграций

### 15.10.1. 1С-клиент-банк

- Импорт — только через загрузку файла (или опрос локальной директории), НЕ через API банка.
- Директория опроса — на сервере, доступ только у пользователя `worker`.

### 15.10.2. Email (IMAP/SMTP)

- Пароль приложения (не основной пароль аккаунта).
- IMAP — только чтение (`READ_ONLY=True` в imapflow).
- SMTP — только отправка, без receiving прав.

### 15.10.3. Telegram

- Bot token — в env var.
- `allowedUsers` — whitelist chatId, остальные игнорируются.
- Команды от незнакомых chatId — логируются как предупреждения.

### 15.10.4. LLM API

- Ключи — в env var.
- Запросы — только с сервера (server-to-server), не с клиента.
- Таймаут 30 секунд на запрос, retry с экспоненциальной задержкой.

## 15.11. Реагирование на инциденты

### 15.11.1. Обнаружение

- Аномалии в логах (большое количество 4xx/5xx ошибок) → алерт администратору.
- Аномалии в API (превышение rate limit, подозрительные паттерны) → временная блокировка.
- Утечка секретов (например, через GitHub) → немедленная ротация.

### 15.11.2. Реагирование

1. Изоляция — отключить скомпрометированный аккаунт.
2. Оценка — что утекло, кому, когда.
3. Уведомление — пострадавшим (если ПД) + Роскомнадзор (в течение 24 часов при утечке ПД).
4. Исправление — закрыть уязвимость.
5. Пост-мортем — документированный разбор, меры на будущее.

- [ ] RBAC проверен на всех эндпоинтах.

- [ ] Логи не содержат sensitive-данных.
- [ ] Бэкапы настроены и протестированы.
