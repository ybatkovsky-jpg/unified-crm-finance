# 01. Введение

## 1.1. Назначение документа

Настоящий документ описывает требования к единой CRM-системе, объединяющей функциональность двух существующих программных комплексов компании — zakuppro и finpro — в один программный продукт. Документ предназначен для:

- **Заказчика / владельца бизнеса** — для подтверждения рамок проекта и приоритетов.
- **Архитектора и тимлида** — для принятия технических решений и декомпозиции на спринты.
- **Разработчиков** — для понимания модулей, сущностей и API-контрактов.
- **QA и аналитиков** — для формирования тест-планов и приёмочных критериев.
- **DevOps и SecOps** — для проектирования инфраструктуры и контроля безопасности.

## 1.2. Область действия

Документ покрывает функциональные и нефункциональные требования к единой системе, целевую архитектуру, модель данных, API-контракты, требования безопасности, дорожную карту на 6 месяцев и реестр рисков. Документ НЕ содержит исходный код реализации, инфраструктурные конфигурации (Docker-compose, Kubernetes-манифесты), детальные тест-планы и пользовательскую документацию — они будут разработаны отдельными артефактами в соответствующих спринтах.

## 1.3. Контекст бизнес-задачи

Компания занимается строительными и монтажными проектами (мебельное производство, оснащение объектов). Текущий ландшафт ПО:

- **zakuppro** — автоматизирует цикл «спецификация → запрос поставщику → счёт → сверка → оплата → склад → производство». Содержит AI-агента для парсинга Excel-спецификаций и сверки счетов, email-воркер (IMAP), Telegram-бота для приёма файлов от владельца, склад с резервированием, Kanban с guardrails. Бэкенд на Python/FastAPI, фронтенд на Next.js.
- **finpro** — финансовый контур с элементами CRM: клиенты, проекты, транзакции, бюджеты, кассовые разрывы, импорт 1С-клиент-банк, синхронизация с zakuppro по API, аудит, RBAC. Полностью на Next.js + Prisma.

Между системами есть односторонняя интеграция (finpro тянет проекты из zakuppro), но нет единой клиентской базы, воронки сделок и сквозного учёта от первого контакта до закрытия проекта.

## 1.4. Цели проекта

### 1.4.1. Бизнес-цели

| ID | Цель | Метрика успеха |
|----|------|----------------|
| BG-01 | Сократить время от первого контакта до создания проекта в системе | С 7 дней (среднее сейчас) до 1 рабочего дня |
| BG-02 | Исключить двойной ввод данных между системами | 0 ручных переносов между системами после миграции |
| BG-03 | Дать руководителю сквозную видимость маржинальности по проекту в реальном времени | Маржинальность видна в дашборде < 5 минут после проводки |
| BG-04 | Автоматизировать согласование заявок на закупку и платежей | Время согласования с 2 дней до 4 часов |
| BG-05 | Повысить конверсию воронки сделок за счёт прозрачности статусов | +15% к конверсии лид→договор за 6 месяцев после запуска |

### 1.4.2. Технические цели

| ID | Цель | Метрика |
|----|------|---------|
| TC-01 | Единая кодовая база | Один репозиторий, одна CI/CD pipeline |
| TC-02 | Единая база данных | Один экземпляр PostgreSQL, одна Prisma-схема |
| TC-03 | Единая аутентификация | Один JWT-токен работает для всех модулей |
| TC-04 | Единый UI/UX | Один дизайн-токен, один набор shadcn-компонентов |
| TC-05 | Покрытие тестами критичных путей | >70% unit-покрытия на бизнес-логику, E2E на все ключевые сценарии |

## 1.5. Границы проекта (Scope)

### 1.5.1. В рамках проекта (In-Scope)

- Проектирование и разработка единой CRM-системы со всеми 8 модулями (см. раздел 1.6).
- Перенос функциональности из zakuppro: закупки, спецификации, склад, AI-агент (парсинг Excel, сверка счетов), email-воркер, Telegram-бот.
- Перенос функциональности из finpro: финансы, транзакции, бюджеты, кассовые разрывы, импорт 1С-клиент-банк, RBAC, аудит.
- Разработка с нуля: модуль CRM (контакты, источники лидов, история взаимодействий), модуль Сделки и воронка, модуль Договоры (генерация по шаблонам, версии, согласование), модуль Аналитика (сквозные дашборды).
- Единая инфраструктура: Docker Compose для dev/staging, продакшен-деплой через CI/CD.
- Документация: пользовательская (краткие руководства по модулям), административная (настройка, бэкапы, мониторинг).

### 1.5.2. Вне рамок (Out-of-Scope)

- Мобильные нативные приложения (iOS/Android). Веб-интерфейс адаптивный и работает в мобильных браузерах.
- Интеграция с внешними CRM (Bitrix24, amoCRM, HubSpot).
- Интеграция с государственными системами (ЕГАИС, Меркурий, Честный ЗНАК, СБИС).
- Поддержка нескольких юридических лиц с раздельным финансовым учётом (multicompany). Рассматривается как backlog-эпик для следующей фазы.
- Data Warehouse для BI-инструментов (Power BI, Metabase). Встроенной аналитики достаточно.
- IP-телефония и интеграция с АТС (запись звонков, автоматическое создание лидов из звонков). Может быть добавлено в backlog.
- Электронная подпись договоров через КЭП (сейчас — скачивание PDF, подписание вручную, загрузка скана).

## 1.7. Глоссарий

| Термин | Определение |
|--------|-------------|
| **BOM** (Bill of Materials) | Спецификация — структурированный список позиций (товаров/материалов) с количеством, необходимый для выполнения проекта. В закупках — основа для формирования заказов поставщикам. |
| **Канбан** | Способ визуализации потока задач/сделок в виде колонок, соответствующих стадиям процесса, с перемещением карточек между ними. |
| **Контрагент** | Юридическое или физическое лицо, выступающее стороной в сделке/договоре. Включает клиентов, поставщиков, подрядчиков. |
| **Lead (лид)** | Потенциальный клиент, проявивший интерес, но ещё не заключивший договор. |
| **Deal (сделка)** | Конкретная потенциальная сделка с клиентом — от первого контакта до подписания договора или отказа. |
| **Project (проект)** | Конкретный объект/заказ, выполняемый компанией после заключения договора. Связан со сделкой «многие-к-одному». |
| **Счёт (invoice)** | Документ от поставщика с указанием позиций, цен и суммы к оплате. |
| **Payment (платёж)** | Фактическое движение денег по банковской выписке или кассе. |
| **Маржинальность** | Отношение прибыли к выручке по проекту: `(contract_amount - total_cost) / contract_amount`. |
| **Кассовый разрыв** | Ситуация, когда плановые платежи превышают доступные денежные средства на период. |
| **RBAC** (Role-Based Access Control) | Модель доступа, при которой права выдаются ролям, а пользователи назначаются на роли. |
| **Сотрудник** | Внутренний пользователь системы с одной или несколькими ролями. |
| **1С-клиент-банк** | Стандартный XML-формат выгрузки банковских операций из 1С, поддерживаемый большинством российских банков. |
| **DLQ** (Dead Letter Queue) | Очередь «мёртвых писем» — куда попадают задачи, которые не удалось обработать после нескольких ретраев. |
| **PoC** (Proof of Concept) | Упрощённая реализация для проверки концепции. |
| **UAT** (User Acceptance Testing) | Приёмочное тестирование с реальными пользователями. |
| **Спринт** | Итерация разработки фиксированной длины (в нашем случае — 2 недели). |

## 1.8. Допущения и зависимости

### 1.8.1. Допущения

- Заказчик предоставляет доступ к production-окружениям zakuppro и finpro для анализа фактических данных и интеграционных точек.
- Команда разработки состоит минимум из 4 человек (1 лид + 2 fullstack + 1 Python/AI), плюс 1 QA/аналитик на part-time.
- Заказчик назначает product owner'а с полномочиями принимать решения по приоритетам в течение 1 рабочего дня.
- Инфраструктура (сервер, домен, SSL, SMTP, Telegram-бот) предоставляется заказчиком в течение первого спринта.
- Тестовые данные для разработки формируются командой на основе обезличенных образцов от заказчика.

### 1.8.2. Зависимости

- Доступ к API банков (или регулярная выгрузка 1С-клиент-банк) для модуля Финансы.
- Доступ к email-ящику (IMAP/SMTP) для email-воркера модуля Закупки.
- Telegram-бот token для модуля Уведомления.
- API-ключи LLM-провайдеров (DeepSeek, OpenAI, Anthropic) для AI-агента.
- Доступ к DNS-записи домена для выпуска SSL-сертификата.

## 1.9. Соответствие стандартам

- **REST API** — следует принципам REST, JSON в качестве формата обмена.
- **OpenAPI 3.1** — спецификация API публикуется в формате OpenAPI, доступна по `/api/docs`.
- **OAuth 2.0 / OIDC** — не используется в первой версии (NextAuth с JWT достаточно), но схема预留 for future.
- **GDPR / 152-ФЗ** — персональные данные клиентов хранятся в РФ-юрисдикции, согласие на обработку фиксируется в карточке контакта.
- **PCI DSS** — неприменимо (платёжные карты не обрабатываются в системе).
