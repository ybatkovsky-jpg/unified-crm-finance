# 16. Нефункциональные требования

## 16.1. Производительность

### 16.1.1. Время отклика API

| Тип запроса | Целевое время | Максимум (95-й перцентиль) |
|-------------|---------------|----------------------------|
| GET списка (до 100 записей) | < 200 мс | 500 мс |
| GET карточки сущности | < 150 мс | 400 мс |
| POST создание сущности | < 300 мс | 800 мс |
| PUT обновление | < 250 мс | 600 мс |
| DELETE (soft) | < 200 мс | 500 мс |
| GET аналитического отчёта (кеш горячего) | < 500 мс | 2 с |
| GET аналитического отчёса (холодный расчёт) | < 3 с | 10 с |
| Загрузка файла (10 МБ) | < 5 с | 15 с |
| Экспорт в Excel (до 10k строк) | < 3 с | 10 с |
| Импорт 1С-выписки (1000 операций) | < 5 с | 15 с |

### 16.1.2. Время загрузки UI

| Метрика | Цель |
|---------|------|
| First Contentful Paint (FCP) | < 1.5 с |
| Largest Contentful Paint (LCP) | < 2.5 с |
| Time to Interactive (TTI) | < 3 с |
| Cumulative Layout Shift (CLS) | < 0.1 |
| Первая загрузка приложения | < 3 с |
| Переход между страницами | < 500 мс |

### 16.1.3. Пропускная способность

- 100 одновременных пользователей — без деградации.
- 50 RPS на API — без деградации.
- Пиковая нагрузка — 200 RPS в течение 5 минут — с допустимой деградацией < 30%.

### 16.4.4. Размер БД

- До 10 млн транзакций — производительность не падает.
- До 1 млн контактов — поиск < 200 мс.
- До 100k проектов — списки < 500 мс.

## 16.2. Доступность

| Метрика | Целевое значение |
|---------|-----------------|
| Uptime в рабочие часы (09:00–20:00 MSK, Пн-Пт) | 99.5% |
| Uptime круглосуточно | 99.0% |
| Плановое обслуживание | 1 раз в месяц, до 2 часов, в вс 02:00–04:00 |
| RTO (после сбоя) | 4 часа |
| RPO (потеря данных) | 1 час (с WAL archiving) |

### 16.2.1. Деградация

При сбое отдельных компонентов:

- **Python-воркер упал** — UI работает, но AI-парсинг BOM и email-приём приостановлены. Показывается баннер «Часть функций недоступна».
- **RabbitMQ упал** — UI работает, но фоновые задачи (импорт, отправка email) в очереди. Восстановление — автоматически после поднятия RabbitMQ.
- **S3/MinIO упал** — UI работает, но загрузка/скачивание файлов недоступно.
- **БД упала** — полная недоступность системы. Alarm администратору.

## 16.3. Масштабируемость

### 16.3.1. Вертикальное

- Увеличение CPU/RAM сервера — до 8 vCPU / 32 ГБ RAM (достаточно для 100 пользователей).
- Увеличение диска БД — до 1 ТБ (достаточно для 5 лет работы).

### 16.3.2. Горизонтальное

- Next.js — stateless, можно запускать несколько инстансов за load balancer.
- Python-воркер — можно запускать несколько инстансов Celery worker.
- PostgreSQL — read replica для аналитических запросов (опционально).

### 16.3.3. Узкие места

- **Сортировка и фильтрация больших списков** — решается индексами и пагинацией.
- **Аналитические запросы** — материализованные представления, кеш.
- **AI-задачи** — ограничение concurrency (не более 5 параллельных LLM-запросов).

## 16.4. Надёжность

### 16.4.1. Идемпотентность

- Все POST/PUT/DELETE с `Idempotency-Key` — гарантия отсутствия дублей.
- Celery-задачи — идемпотентны по `externalId` или `taskId`.
- Webhook-обработчики — идемпотентны (повторная обработка не создаёт дублей).

### 16.4.2. Транзакционность

- Все мутации, затрагивающие несколько таблиц — в одной Prisma-транзакции (`$transaction`).
- Списание со склада + создание `WarehouseTransaction` — атомарно.
- Создание `Transaction` + обновление `Invoice.status` — атомарно.

### 16.4.3. Retry и DLQ

- Celery-задачи: 3 попытки с экспоненциальной задержкой (1, 5, 30 минут).
- После 3 неудач — задача в DLQ, уведомление администратору.
- DLQ-админка — ручной перезапуск или пометка как «обработано вручную».

### 16.4.4. Circuit breaker

- Для внешних API (LLM, email, банк) — circuit breaker: при 5 подряд ошибках — пауза 5 минут, потом пробный запрос.

## 16.5. Совместимость

### 16.5.1. Браузеры

- Chrome 100+, Firefox 100+, Safari 15+, Edge 100+.
- Mobile: iOS Safari 15+, Android Chrome 100+.
- Internet Explorer — не поддерживается.

### 16.5.2. Разрешения экрана

- Desktop: от 1280×720.
- Планшет: от 768×1024 (адаптивный UI).
- Мобильный: от 375×667 (упрощённый UI для ключевых функций).

### 16.5.3. API-клиенты

- Любой HTTP-клиент, поддерживающий JSON.
- Официальные SDK — не предоставляются в первой версии (только OpenAPI-спецификация).

## 16.6. Локализация

### 16.6.1. Языки

- Русский — основной, по умолчанию.
- Английский — добавляется в следующей фазе (фреймворк next-intl уже заложен).

### 16.6.2. Форматирование

- Даты: `DD.MM.YYYY` (русский формат).
- Время: 24-часовой формат, `HH:MM`.
- Числа: десятичная точка — запятая в русском, разделитель тысяч — пробел (`1 234 567,89`).
- Валюты: `RUB` по умолчанию, символ `₽`, формат `1 234 567 ₽`.
- Часовой пояс: MSK (UTC+3) по умолчанию, настраивается на уровне системы.

### 16.6.3. Кодировка

- Все данные — UTF-8.
- Поддержка всех кириллических символов, эмодзи в notes (но не в служебных полях).

## 16.7. Управляемость

### 16.7.1. Мониторинг

- Health endpoints: `/api/health` (Next.js), `/internal/health` (Python).
- Метрики в Prometheus-формате: `/api/metrics` (только для мониторинга, защищён).
- Логи — JSON в stdout, собираются Docker logs driver (или в ELK/CloudWatch при необходимости).

### 16.7.2. Алерты

- Внешний мониторинг (Uptime Robot или аналог) — пинг `/api/health` раз в минуту.
- При недоступности >2 минут — SMS/email администратору.
- Метрики БД (CPU, диск, connections) — алерты при 80% утилизации.

### 16.7.3. Бэкапы и восстановление

- Ежедневный бэкап БД — в 03:00 MSK.
- Проверка restore — еженедельно на staging.
- Документация по восстановлению — в `docs/runbook/restore.md` (будет написана в спринте 12).

### 16.7.4. Конфигурация

- Все настройки — через env vars.
- `.env.example` в репозитории — с описанием каждой переменной.
- Runtime-настройки (через UI): настройки email, LLM, Telegram — в таблице `Setting`.

## 16.9. Юзабилити

### 16.9.1. Принципы

- Consistency — единый стиль компонентов (shadcn/ui).
- Feedback — каждое действие пользователя имеет визуальный отклик (<200 мс).
- Error prevention — валидация на клиенте перед отправкой, guardrails на опасные действия.
- Error recovery — понятные сообщения об ошибках, предложения по исправлению.
- Keyboard friendly — все основные действия доступны с клавиатуры (command palette на Ctrl+K).
- Accessibility — WCAG 2.1 AA.

### 16.9.2. Производительность восприятия

- Skeleton loaders для списков и карточек.
- Optimistic updates для форм (с откатом при ошибке).
- Prefetch связанных данных при наведении на ссылки.
- Lazy loading для тяжёлых компонентов (Markdown editor, charts).

## 16.10. Экологические аспекты

Не применимо напрямую (программный продукт), но:

- Оптимизация запросов к LLM — меньше токенов = меньше энергозатрат на обучение/вывод.
- Кеширование аналитики — меньше нагрузки на БД.
- Эффективные алгоритмы — O(n) вместо O(n²) где возможно.

## 16.11. Юридические аспекты

- Лицензия на ПО — проприетарная (см. LICENSE).
- Персональные данные — 152-ФЗ, согласие в карточке контакта.
- Коммерческая тайна — NDA с сотрудниками (вне системы).
