# 06. Модуль CRM и контакты

## 6.1. Назначение

Модуль CRM — единая точка входа для всех контактов с внешним миром. Любой, кто позвонил, зашёл в офис, написал на email или в Telegram, обращается через сайт или пришёл по рекомендации, должен быть заведён в систему как `Contact`. Это позволяет:

- Иметь единую базу клиентов, лидов, контрагентов (без дубликатов).
- Отслеживать источник каждого контакта (для оценки эффективности каналов).
- Вести историю всех взаимодействий с контактом.
- Передавать контакты в воронку продаж (создавать `Deal`).
- Использовать контакты при создании договоров и проектов.

## 6.2. Функциональные требования

### 6.2.1. Управление контактами

| ID | Требование | Приоритет |
|----|------------|-----------|
| CRM-01 | Система должна позволять создавать контакт типа `person` (физлицо) с полями: фамилия, имя, отчество, телефон, email, адрес, заметки | Must |
| CRM-02 | Система должна позволять создавать контакт типа `company` (юрлицо) с полями: наименование, ИНН, КПП, ОГРН, юридический адрес, фактический адрес, телефон, email, контактное лицо | Must |
| CRM-03 | Система должна искать дубликаты при создании контакта: по ИНН (для company), по телефону, по email. При обнаружении — показывать существующий контакт с предложением слить | Must |
| CRM-04 | Система должна позволять объединять дубликаты контактов с переносом всех связанных данных (сделок, взаимодействий, договоров, проектов) на выбранный «основной» контакт | Must |
| CRM-05 | Система должна позволять назначать ответственного менеджера (owner) на контакт | Must |
| CRM-06 | Система должна позволять тегировать контакты произвольными тегами (например, «VIP», «оптовик», «г. Москва») | Should |
| CRM-07 | Система должна хранить произвольные дополнительные атрибуты контакта в JSON-поле `attributes` (для кастомных полей без изменения схемы) | Should |
| CRM-08 | Система должна поддерживать soft-delete контакта (при удалении — `deletedAt`, не физическое) с возможностью восстановления | Must |
| CRM-09 | Система должна блокировать удаление контакта, у которого есть активная сделка или договор (только пометить `archived`) | Must |
| CRM-10 | Система должна экспортировать список контактов в Excel (с фильтрами, применёнными в UI) | Should |

### 6.2.2. Источники лидов

| ID | Требование | Приоритет |
|----|------------|-----------|
| CRM-11 | Система должна иметь предзагруженный справочник источников лидов: звонок, визит в офис, сайт, email, Telegram, рекомендация, другое | Must |
| CRM-12 | Администратор должен иметь возможность добавлять кастомные источники лидов | Should |
| CRM-13 | Каждый контакт должен иметь проставленный источник (обязательное поле при создании) | Must |
| CRM-14 | Система должна показывать отчёт «конверсия по источникам» — сколько контактов из каждого источника стали клиентами | Should |

### 6.2.3. Взаимодействия

| ID | Требование | Приоритет |
|----|------------|-----------|
| CRM-15 | Система должна позволять фиксировать взаимодействия с контактом: звонок, встреча, email, сообщение в мессенджере, визит, заметка | Must |
| CRM-16 | Каждое взаимодействие должно иметь тип, направление (для звонков/email — входящее/исходящее), тему, содержание, дату, автора | Must |
| CRM-17 | Система должна позволять планировать будущие взаимодействия (со `scheduledAt`, без `completedAt`) с напоминанием | Must |
| CRM-18 | Система должна показывать в карточке контакта таймлайн всех взаимодействий в обратном хронологическом порядке | Must |
| CRM-19 | Система должна показывать в дашборде менеджера список запланированных на сегодня/завтра взаимодействий | Must |
| CRM-20 | Система должна отправлять уведомление менеджеру за 15 минут до запланированного взаимодействия | Should |

### 6.2.4. Поиск и фильтрация

| ID | Требование | Приоритет |
|----|------------|-----------|
| CRM-21 | Система должна искать контакты по любому полю (имя, ИНН, телефон, email, компания) из единой строки поиска | Must |
| CRM-22 | Система должна поддерживать фильтрацию списка контактов: по типу, источнику, тегам, ответственному, статусу, дате создания | Must |
| CRM-23 | Система должна поддерживать сортировку по: дате создания, дате обновления, имени, последнему взаимодействию | Must |
| CRM-24 | Система должна использовать PostgreSQL full-text search по полям `firstName`, `lastName`, `companyName`, `notes` для быстрого поиска | Should |

## 6.3. User Stories

- **US-CRM-01**: Как менеджер по продажам, я хочу видеть на одной странице всех, кто обратился к нам за последний месяц, с указанием источника, чтобы оценить эффективность каналов.
- **US-CRM-02**: Как менеджер, я хочу при звонке от нового клиента быстро завести его в систему, указав имя, телефон и источник «звонок», чтобы не потерять лид.
- **US-CRM-03**: Как владелец, я хочу видеть, какие контакты без ответственного менеджера «висят» дольше 3 дней, чтобы перераспределить их.
- **US-CRM-04**: Как менеджер, я хочу в карточке клиента видеть всю историю: когда звонил, о чём договорились, какие КП отправлял, чтобы не переспрашивать.
- **US-CRM-05**: Как менеджер, я хочу планировать повторный звонок через неделю с напоминанием, чтобы не забыть «дожать» клиента.

## 6.4. UI/UX

### 6.4.1. Список контактов

- Таблица с колонками: Имя/Компания, Тип, Телефон, Email, Источник, Ответственный, Теги, Последнее взаимодействие, Создан.
- Фильтры сверху: тип, источник, ответственный, теги, дата создания, статус.
- Поиск по строке (top-right).
- Кнопка «Создать контакт» (top-right).
- Контекстное меню на строке: Открыть, Создать сделку, Создать взаимодействие, Объединить с..., Экспорт в Excel.
- Pagination по 50/100/200 записей.

### 6.4.2. Карточка контакта

Шапка:
- Аватар/иконка (для person — инициалы, для company — иконка).
- Имя/Наименование крупно.
- Тип бейджем (Физлицо/Юрлицо).
- Телефон, email — кликабельны (tel:/mailto:).
- Кнопки: Создать сделку, Создать взаимодействие, Редактировать, Удалить.

Табы:
1. **Обзор** — основные поля, ответственный, теги, источник, заметки.
2. **Взаимодействия** — таймлайн (как в Activity log).
3. **Сделки** — список связанных сделок со статусами.
4. **Договоры** — список связанных договоров.
5. **Проекты** — список связанных проектов.
6. **Финансы** — сумма поступлений от этого контакта, сумма долгов.
7. **История изменений** — audit log.

### 6.4.3. Форма создания/редактирования

- Переключатель типа: Физлицо / Юрлицо (меняет набор полей).
- Для физлица: Фамилия*, Имя*, Отчество, Телефон*, Email, Адрес, Заметки.
- Для юрлица: Наименование*, ИНН*, КПП, ОГРН, Юр. адрес, Факт. адрес, Телефон*, Email, Контактное лицо, Заметки.
- Источник* (select).
- Ответственный (select из пользователей с ролью sales/manager).
- Теги (multi-select с возможностью добавления новых).
- Кнопка «Сохранить» / «Отмена».
- При сохранении — проверка дублей (по ИНН для юрлица, по телефону для физлица). Если найден — модал «Найден похожий контакт, открыть его? Да / Всё равно создать».

```

POST   /api/v1/contacts/:id/restore      — восстановление
POST   /api/v1/contacts/merge            — объединение { primaryId, secondaryId }
GET    /api/v1/contacts/:id/interactions — список взаимодействий

GET    /api/v1/contacts/:id/deals        — связанные сделки
```

## 6.7. Интеграции с другими модулями

- **Sales**: при создании сделки — обязательная ссылка на `Contact`. В сделке показывается мини-карточка контакта.
- **Contracts**: при создании договора — выбор клиента из `Contact` (тип company обязателен, ИНН — в шаблон).
- **Projects**: проект наследует `contactId` от сделки или создаётся напрямую с контактом.
- **Finance**: поступления от клиента (`Transaction.income` с `counterpartyId`) агрегируются в карточке контакта → «Всего поступило».
- **Notifications**: если контакт без взаимодействий >14 дней — уведомление ответственному менеджеру.

## 6.8. Метрики модуля

- Количество новых контактов за период (по источникам).
- Количество активных контактов (сделки в работе).
- Среднее время от первого контакта до создания сделки.
- Доля «брошенных» контактов (без взаимодействий >30 дней).
- Конверсия контакты → сделки → договоры по источникам.
