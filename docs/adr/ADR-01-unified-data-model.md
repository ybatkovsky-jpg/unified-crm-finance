# ADR-01: Унификация моделей данных zakuppro и finpro

- **Статус:** Proposed (ожидает ревью заказчика)
- **Дата:** 2026-06-19
- **Автор:** Spec Bot
- **Решает:** спринт 1, эпик S1-06 / S1-10

## Контекст

zakuppro и finpro — два независимо разработанных продукта с пересекающейся, но не идентичной моделью данных:

| Сущность | zakuppro | finpro | Расхождение |
|----------|----------|--------|-------------|
| Поставщик / контрагент | `Supplier` (Prisma + SQLAlchemy) | `Counterparty` (Prisma) | Дублирование; в finpro есть `type=supplier|customer|other`, в zakuppro — без типа |
| Клиент | `customerName` строкой в `Project` | `Client` (отдельная таблица, 5 полей) | finpro богаче, но не хватает для CRM |
| Проект | `Project` (без `clientId`, `customerName` строкой) | `Project` (с `externalId`, `clientId`, `marginTarget`, `qualityRating`) | finpro богаче; нужен гибрид |
| Позиция спецификации | `ProjectItem` (привязан к Project напрямую) | нет | нужен уровень `BOM` для версионности |
| Транзакция | нет (только `Invoice` + `Payment` через bank worker) | `Transaction` (полная, с дедупликацией) | finpro — основа |
| Категория | нет | `Category` (иерархия) | finpro — основа |
| Бюджет | нет | `Budget` | finpro — основа |
| Склад | `WarehouseItem` + `WarehouseTransaction` | нет | zakuppro — основа |
| Запрос поставщику | `PurchaseRequest` + `PurchaseRequestItem` | нет | zakuppro — основа |
| Счёт | `Invoice` + `InvoiceItem` | нет | zakuppro — основа |
| Email-лог | `EmailLog` | нет | zakuppro — основа |
| Поставка | `Delivery` | нет | zakuppro — основа |
| Настройки | `EmailSettings`, `AiSettings`, `TelegramSettings`, `CompanyDetails` | нет | упрощаем до `Setting` |
| Пользователь | (auth в Python) | `User` (Prisma, 4 роли) | расширяем до many-to-many `User`+`Role`+`UserRole` |
| Аудит | `ProjectStatusHistory` (только статусы проекта) | `AuditLog` (общий, с diff JSON) | finpro — основа, расширяем на все сущности |

Без унификации мы получили бы:
- Дублирующихся контрагентов (поставщик как `Supplier` и как `Counterparty`).
- Несогласованные ID-форматы (uuid в finpro, cuid в zakuppro).
- Расходящиеся схемы при эволюции (Prisma-схема и SQLAlchemy-модели легко «разъезжаются»).
- Сложные JOIN'ы между двумя источниками правды.

## Решение

### 1. Единый источник правды — Prisma

`apps/web/prisma/schema.prisma` — единственное место, где описывается структура БД. Все таблицы создаются миграциями Prisma (`prisma migrate`).

Python-воркер подключается к той же БД, но **не создаёт таблицы и не запускает свои миграции для таблиц**. SQLAlchemy используется в режиме `reflect=True` (читает схему из БД) или с явными моделями только для чтения.

### 2. Унифицированный набор сущностей (40+)

Полный список и поля — в [`docs/05-data-model.md`](../05-data-model.md). Ключевые слияния:

| Что | Как | Обоснование |
|-----|-----|-------------|
| `Supplier` (zakuppro) + `Counterparty` (finpro) | → единый `Counterparty` с полем `types[]` (supplier/customer/contractor/other) | Один контрагент может быть и поставщиком, и клиентом — это нормальная практика |
| `Client` (finpro) + новый CRM | → `Contact` с `type=person|company` и расширенными полями (ИНН, КПП, ОГРН, адреса) | Единая база всех контактов с внешним миром |
| `Project` (обеих систем) | → единый `Project` с расширенными полями: `dealId`, `contractId`, `contactId`, `marginTarget`, `qualityRating`, `deadlineStatus` | Гибрид лучших полей из обеих систем |
| `ProjectItem` (zakuppro) | → `BOMItem` (через `BOM` для версионности) | Добавлен уровень `BOM` для поддержки версий спецификации |
| `ProjectStatusHistory` (zakuppro) | → общий `AuditLog` (как в finpro) | Единый журнал для всех сущностей, не только проектов |
| `EmailSettings`/`AiSettings`/`TelegramSettings`/`CompanyDetails` (zakuppro) | → единая таблица `Setting` (key-value с JSON) | Упрощение: не нужны 4 разные таблицы для настроек |
| `User` (finpro, 4 роли в одном поле) | → `User` + `Role` + `UserRole` (many-to-many) | Один пользователь может иметь несколько ролей (например, manager + accountant) |

### 3. ID-стратегия

- Все первичные ключи — `UUID` (PostgreSQL `uuid` type, генерация `gen_random_uuid()` через `@default(uuid())` в Prisma).
- У внешних интеграций — `externalId` поле (например, `Transaction.externalId` для 1С-выписки, `Project.externalNumber` для ПМ-номеров).
- Уникальные человекочитаемые номера — отдельные поля: `Deal.number` (С-2026-00001), `Contract.number` (Д-2026-00001), `Project.externalNumber` (ПМ-2026-00001).

### 4. Связь между bounded contexts

Связи между модулями — через `*Id` поля, **без FK-ограничений на уровне БД** (только индексы). Это позволяет:

- Контекстам развиваться независимо (например, можно временно отключить модуль Закупок, не ломая Финансы).
- Гибко менять ссылки (например, перевести сделку на другой контакт без каскадных обновлений).
- Проверка целостности — на уровне приложения (zod-схемы + Prisma middleware).

Исключение: связи внутри одного контекста (например, `BOMItem.bomId` → `BOM.id`) — с FK и `onDelete: Cascade`.

### 5. Soft delete

Для критичных сущностей (`Contact`, `Deal`, `Contract`, `Project`, `Transaction`) — поле `deletedAt`. Запросы по умолчанию фильтруют `deletedAt IS NULL`. Восстановление — обнулением поля. Полное физическое удаление — только через админский скрипт с двойным подтверждением.

### 6. JSONB для динамических атрибутов

Поля `attributes Json?` на `Contact`, `Deal`, `Project`, `Counterparty` — для кастомных полей без миграций. Например, дополнительные реквизиты клиента, отраслевые теги. Структура JSON не валидируется на уровне БД, только на уровне приложения (zod).

### 7. Аудит

Единый `AuditLog` для всех сущностей (не только статусов проекта, как в zakuppro):
- `entityType` + `entityId` — над чем действие.
- `action` — `create|update|delete|restore|login|logout|sync`.
- `changes` — JSON `{ before: {...}, after: {...} }`, только изменённые поля.
- `metadata` — JSON `{ ip, userAgent, source }`.

Заполняется через Prisma middleware (автоматически на всех мутациях) + явные вызовы в сервисах.

## Альтернативы, рассмотренные и отвергнутые

### Альтернатива A: Микросервисы с отдельными БД

Каждый модуль — отдельный сервис со своей БД, синхронизация через события.

**Почему нет:**
- Размер команды (4–6 человек) не тянет микросервисы.
- Транзакционная консистентность между CRM, закупками и финансами критична (нельзя допустить, чтобы договор создался в одном сервисе, а проект — нет).
- DevOps-накладные расходы.

### Альтернатива B: Оставить две БД, синхронизировать через API

Как сейчас в finpro: pull-синхронизация проектов из zakuppro.

**Почему нет:**
- Двойной ввод данных, который мы и хотим устранить.
- Сложность поддержки консистентности.
- Невозможность сквозных запросов (например, P&L по проекту со спецификацией).

### Альтернатива C: SQLAlchemy как основной ORM, Prisma убрать

**Почему нет:**
- Prisma даёт типизацию end-to-end с TypeScript-фронтендом, что ускоряет разработку.
- Prisma middleware удобнее для аудита.
- Команда больше знакома с TypeScript (судя по коду finpro).

### Альтернатива D: Drizzle ORM вместо Prisma

**Почему нет:**
- Drizzle моложе, меньше экосистема.
- В исходных проектах уже используется Prisma — меньше переписывания.

## Последствия

### Положительные

- Единая схема, легко поддерживать и эволюционировать.
- Типизация end-to-end (Prisma → TypeScript → React).
- Аудит всех действий в одном месте.
- Сквозные запросы через JOIN'ы (без интеграций).

### Отрицательные

- Python-воркер ограничен: нельзя менять схему через Alembic (только через Prisma). Для view-функций и материализованных представлений — отдельный набор SQL-миграций в `apps/web/prisma/sql/`, применяемых после Prisma-миграций.
- Если Prisma не поддержит какой-то PostgreSQL-конструкт (например, сложный RLS) — придётся добавлять raw SQL в миграции.
- Размер Prisma-схемы (40+ моделей) может стать неудобным; план — разбить на несколько `.prisma` файлов через `prisma-schema-multi` (опционально, если станет проблемой).

## План реализации

1. **S1 (недели 1–2):** написать `apps/web/prisma/schema.prisma` по [`docs/05-data-model.md`](../05-data-model.md), создать первые миграции, проверить что Prisma + SQLAlchemy видят одну схему.
2. **S2 (недели 3–4):** Python-воркер подключается к БД через SQLAlchemy `reflect=True`, пишет тестовые записи, читает их через Next.js.
3. **На каждом спринте:** CI-проверка — `prisma migrate diff` между схемой и БД пустой, SQLAlchemy-модели валидны.
