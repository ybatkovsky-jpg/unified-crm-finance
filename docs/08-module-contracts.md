# 08. Модуль Договоры

## 8.1. Назначение

Модуль Договоры автоматизирует процесс создания, согласования и хранения договорных документов. Из сделки (или напрямую) создаётся договор по шаблону, генерируется PDF, при необходимости — проходится цикл согласования, после подписания (загрузки скана) автоматически создаётся проект.

Цели модуля:
- Унификация договорной базы (все договоры в одном месте, с версионностью).
- Сокращение времени на подготовку договора с 30 минут до 2 минут (за счёт шаблонов).
- Исключение ошибок ручного ввода (реквизитов, сумм) через автоподстановку из контакта и сделки.
- Сквозная связь: сделка → договор → проект.

## 8.2. Функциональные требования

### 8.2.1. Шаблоны договоров

| ID | Требование | Приоритет |
|----|------------|-----------|
| CONTRACT-01 | Система должна позволять создавать шаблоны договоров (`ContractTemplate`) в формате Markdown с плейсхолдерами `{{clientName}}`, `{{inn}}`, `{{amount}}`, `{{startDate}}` и т.д. | Must |
| CONTRACT-02 | Каждый шаблон должен иметь описание ожидаемых переменных (имя, тип, дефолт) в поле `variables` (JSON) | Must |
| CONTRACT-03 | Система должна предоставлять редактор Markdown с подсветкой и подсказкой плейсхолдеров (на базе @mdxeditor/editor) | Must |
| CONTRACT-04 | Система должна позволять создавать несколько версий шаблона (поле `version`), активная версия — последняя с `isActive=true` | Should |
| CONTRACT-05 | Система должна иметь предзагруженные шаблоны: «Договор поставки», «Договор оказания услуг», «Спецификация к договору» | Must |
| CONTRACT-06 | Администратор должен иметь возможность preview шаблона с тестовыми данными перед сохранением | Should |

### 8.2.2. Создание договора

| ID | Требование | Приоритет |
|----|------------|-----------|
| CONTRACT-07 | Система должна позволять создавать договор: из сделки (кнопка «Создать договор»), или напрямую с указанием контакта | Must |
| CONTRACT-08 | При создании из сделки — автоподстановка: контакт, сумма, ожидаемые даты из сделки | Must |
| CONTRACT-09 | Система должна автоматически генерировать номер договора в формате `Д-2026-00001` | Must |
| CONTRACT-10 | Поля договора: номер, заголовок, клиент (контакт типа company), шаблон, сумма, валюта, дата начала, дата окончания, дата подписания, статус, заметки | Must |
| CONTRACT-11 | Система должна валидировать, что клиент — это контакт типа `company` (для физлиц — упрощённая форма) | Must |
| CONTRACT-12 | При создании договора из сделки — сделка автоматически переводится на выигрышную стадию и получает `contractId` | Must |

### 8.2.3. Генерация PDF

| ID | Требование | Приоритет |
|----|------------|-----------|
| CONTRACT-13 | Система должна генерировать PDF из шаблона с подставленными значениями (через pdfkit на сервере) | Must |
| CONTRACT-14 | Сгенерированный PDF должен сохраняться в S3 и регистрироваться в `FileEntity`, ссылка — в `ContractVersion.generatedPdfFileId` | Must |
| CONTRACT-15 | Система должна создавать новую `ContractVersion` при каждой генерации PDF (история изменений) | Must |
| CONTRACT-16 | Система должна позволять скачивать PDF из карточки договора | Must |
| CONTRACT-17 | PDF должен иметь стандартные элементы: колонтитулы, нумерация страниц, шрифт Times New Roman / Noto Serif (для русского) | Must |
| CONTRACT-18 | Система должна поддерживать подстановку подписантов (ФИО, должность) из `ContractSigner` | Must |

### 8.2.5. Жизненный цикл

| ID | Требование | Приоритет |
|----|------------|-----------|
| CONTRACT-19 | Статусы договора: `draft` → `sent` → `signed` → `active` → `completed` → `cancelled` / `expired` | Must |
| CONTRACT-20 | Переход `draft → sent` — ручной (после отправки клиенту). Переход `sent → signed` — после загрузки скана подписанного. `signed → active` — после указания даты начала. | Must |
| CONTRACT-21 | Система должна требовать загрузку скана подписанного договора (`signedFileId`) для перехода в `signed` | Must |
| CONTRACT-22 | Система должна позволять загрузить несколько версий скана (если договор переподписан), с историей | Should |
| CONTRACT-23 | При переходе в `active` — предлагать создать проект (`Project`) из договора | Must |
| CONTRACT-24 | При создании проекта из договора — `Project.contractId` проставляется, `Project.contactId` наследуется от договора | Must |
| CONTRACT-25 | Система должна уведомлять о приближении `endDate` (за 14 и за 7 дней) — ответственного менеджеру | Should |
| CONTRACT-26 | Система должна помечать договор `expired` автоматически на следующий день после `endDate` (если статус `active`) | Should |

### 8.2.6. Согласование (опционально)

| ID | Требование | Приоритет |
|----|------------|-----------|
| CONTRACT-27 | Система должна позволять настраивать цепочку согласования договора (например, юрист → владелец) | Should |
| CONTRACT-28 | Каждый согласующий должен иметь возможность «Одобрить» или «Отправить на доработку» с комментарием | Should |
| CONTRACT-29 | После всех одобрений — статус `sent`, можно отправлять клиенту | Should |

## 8.3. User Stories

- **US-CONTRACT-01**: Как менеджер, я хочу из карточки сделки нажать «Создать договор», выбрать шаблон, и получить готовый PDF с подставленными реквизитами клиента и суммой.
- **US-CONTRACT-02**: Как менеджер, я хочу загрузить скан подписанного договора, чтобы система автоматически создала проект.
- **US-CONTRACT-03**: Как юрист, я хочу редактировать шаблоны договоров через редактор Markdown, чтобы добавлять новые условия.
- **US-CONTRACT-04**: Как владелец, я хочу видеть все договоры в работе, отсортированные по дате окончания, чтобы планировать продления.
- **US-CONTRACT-05**: Как менеджер, я хочу видеть историю версий договора (какие изменения вносились в текст до подписания), чтобы понимать эволюцию условий.

## 8.4. UI/UX

### 8.4.1. Список договоров

- Таблица: Номер, Заголовок, Клиент, Сумма, Дата подписания, Дата окончания, Статус, Ответственный.
- Фильтры: статус, клиент, ответственный, дата подписания.
- Кнопка «Создать договор» (выбор: из сделки / новый).

### 8.4.2. Карточка договора

Шапка:
- Номер, заголовок, статус (бейдж с цветом).
- Кнопки: Скачать PDF, Загрузить скан, Создать проект, Редактировать, Удалить.

Табы:
1. **Обзор** — основные поля, реквизиты клиента, сумма, даты, подписанты.
2. **Версии** — список версий (Markdown-исходник + PDF), возможность откатиться.
3. **Сканы** — загруженные сканы подписанных документов.
4. **Связи** — сделка, проект, транзакции.
5. **История** — audit log изменений.

### 8.4.3. Создание договора

Шаг 1: выбор источника
- Из сделки (select) — автоподстановка данных.
- С нуля — выбор контакта вручную.

Шаг 2: заполнение полей
- Шаблон* (select).
- Заголовок* (auto из шаблона, можно править).
- Сумма* (auto из сделки).
- Дата начала, дата окончания (date pickers).
- Подписанты с нашей стороны (select из пользователей) и со стороны клиента (ввод ФИО + должности).
- Заметки.

Шаг 3: preview
- Отображение Markdown с подставленными значениями.
- Кнопка «Сгенерировать PDF».

Шаг 4: генерация
- Backend генерирует PDF, сохраняет в S3.
- Создаётся `ContractVersion` с указанием шаблона и значений.
- Статус договора → `draft`.
- Показывается ссылка на скачивание PDF.

## 8.5. API-эндпоинты

```
GET    /api/v1/contract-templates            — список шаблонов
POST   /api/v1/contract-templates            — создание (admin)
PUT    /api/v1/contract-templates/:id        — редактирование (admin)
DELETE /api/v1/contract-templates/:id        — удаление (если нет договоров)
GET    /api/v1/contract-templates/:id/preview — preview с тестовыми данными

GET    /api/v1/contracts                     — список (с фильтрами)
POST   /api/v1/contracts                     — создание
GET    /api/v1/contracts/:id                 — карточка
PUT    /api/v1/contracts/:id                 — обновление полей
DELETE /api/v1/contracts/:id                 — soft delete
POST   /api/v1/contracts/:id/generate-pdf    — генерация PDF (создаёт новую версию)
POST   /api/v1/contracts/:id/upload-scan     — загрузка скана (multipart/form-data)
POST   /api/v1/contracts/:id/sign            — переход в signed { signedAt, signerId }
POST   /api/v1/contracts/:id/activate        — переход в active
POST   /api/v1/contracts/:id/convert-to-project — создание проекта из договора
GET    /api/v1/contracts/:id/versions        — список версий
GET    /api/v1/contracts/:id/versions/:vid/download — скачивание PDF версии
```

## 8.7. Шаблонизатор

### 8.7.1. Синтаксис плейсхолдеров

```
{{ variableName }}                 — простая подстановка
{{ variableName | format:date }}   — с фильтром форматирования
{{#if condition}}...{{/if}}         — условное включение
{{#each items}}...{{/each}}         — цикл
```

### 8.7.3. Сумма прописью

Реализация на TypeScript (отдельный утилитарный модуль `lib/numberToWords.ts`). Поддержка рублей и копеек, родов (рубль/рубля/рублей), правильного склонения. Аналог для других валют — через конфигурацию.

## 8.8. Метрики модуля

- Количество договоров в работе по статусам.
- Среднее время от создания до подписания (дни).
- Сумма договоров за период (с подписанных).
- Доля договоров, созданных из сделок (vs. созданных напрямую).
- Количество договоров, истекающих в ближайшие 30 дней.
- ТОП-5 самых длительных согласований.
