# 13. Модуль Уведомления и согласования

## 13.1. Назначение

Модуль Уведомления — единый центр коммуникации системы с пользователем: in-app уведомления, email-пуши, Telegram-бот, задачи по согласованию (заявки на закупку/оплату/списание). Это «клей», который связывает все остальные модули: любое важное событие в системе должно вызывать уведомление соответствующему пользователю.

## 13.2. Функциональные требования

### 13.2.1. In-app уведомления

| ID | Требование | Приоритет |
|----|------------|-----------|
| NTF-01 | Система должна показывать уведомления в шапке приложения (иконка с бейджем количества непрочитанных) | Must |
| NTF-02 | При клике на иконку — dropdown со списком последних 10 уведомлений | Must |
| NTF-03 | Каждое уведомление: тип (иконка), заголовок, текст, время, ссылка (опц.) | Must |
| NTF-04 | Уведомления можно отметить прочитанными (по одному и массово) | Must |
| NTF-05 | Система должна показывать отдельную страницу «Все уведомления» с фильтрами | Must |
| NTF-06 | Уведомления старше 30 дней автоматически архивируются | Should |
| NTF-07 | Система должна поддерживать real-time обновления через WebSocket (или polling каждые 30 секунд как fallback) | Should |

### 13.2.2. Каналы доставки

| ID | Требование | Приоритет |
|----|------------|-----------|
| NTF-08 | Каждое уведомление должно иметь канал доставки: in-app, email, telegram — настраивается по типу | Must |
| NTF-09 | Пользователь должен иметь возможность настроить, какие уведомления получать на какой канал | Must |
| NTF-10 | Система должна отправлять email через Python-воркер (aiosmtplib) | Must |
| NTF-11 | Система должна отправлять Telegram-сообщения через Python-воркер (python-telegram-bot) | Must |
| NTF-12 | Если пользователь не имеет Telegram ID в профиле — Telegram-канал недоступен | Must |
| NTF-13 | Система должна повторять отправку при ошибке (3 попытки с задержкой 5/30/120 минут) | Must |
| NTF-14 | Неудачные отправки пишутся в `SyncLog` для администратора | Must |

| Код типа | Описание | Кому | Канал по умолчанию |

| `project_deadline` | Дедлайн проекта через 7 дней | Менеджер + owner | in-app + email |
| `project_overdue` | Проект просрочен | Owner | in-app + email + telegram |
| `bom_parsed` | Спецификация распознана AI | Менеджер проекта | in-app |
| `invoice_received` | Получен счёт от поставщика | Закупщик | in-app |
| `invoice_verified` | Счёт сверен | Менеджер | in-app |
| `invoice_discrepancy` | Расхождения в счёте | Менеджер | in-app + email |
| `approval_request` | Заявка на согласование | Согласующий | in-app + email + telegram |
| `approval_decision` | Решение по заявке | Инициатор | in-app + email |
| `payment_received` | Поступила оплата от клиента | Менеджер + owner | in-app + email |
| `payment_made` | Платёж поставщику исполнен | Менеджер | in-app |

### 13.2.4. Согласования (ApprovalRequest)

| ID | Требование | Приоритет |
|----|------------|-----------|
| NTF-15 | Система должна поддерживать типы заявок: `purchase` (закупка), `payment` (платёж), `writeoff` (списание) | Must |
| NTF-16 | Заявка содержит: тип, связанная сущность (PurchaseRequest/Invoice/WarehouseTransaction), сумма, инициатор, статус, дата запроса, дата решения, комментарий | Must |
| NTF-17 | Статусы: `pending` → `approved` / `rejected` / `cancelled` | Must |
| NTF-18 | Инициатор может отозвать заявку (`cancelled`) до решения | Must |
| NTF-19 | Согласующий может одобрить или отклонить с комментарием | Must |
| NTF-20 | После одобрения — выполняется соответствующее действие (создаётся закупка, проводится платёж, списание) | Must |
| NTF-21 | Система должна поддерживать цепочку согласующих (например, manager → owner для сумм >100k) | Should |
| NTF-22 | Система должна показывать список заявок с фильтрами по статусу, типу, инициатору, согласующему | Must |
| NTF-23 | После 3 дней без решения — повторное уведомление согласующему | Must |

### 13.2.5. Telegram-бот

## 13.3. User Stories

- **US-NTF-01**: Как менеджер, я хочу получать уведомление, когда моя сделка перешла на новую стадию (если это сделал кто-то другой), чтобы быть в курсе.
- **US-NTF-02**: Как владелец, я хочу получать Telegram-уведомление о критичных событиях (кассовый разрыв, просрочка проекта), чтобы быстро реагировать.
- **US-NTF-03**: Как владелец, я хочу видеть все заявки на согласование в одном месте, чтобы быстро одобрять или отклонять.
- **US-NTF-04**: Как менеджер, я хочу настроить, какие уведомления получать на email, а какие только in-app, чтобы не тонуть в письмах.
- **US-NTF-05**: Как закупщик, я хочу получить уведомление, что счёт от поставщика сверен, чтобы создать заявку на оплату.

## 13.4. UI/UX

### 13.4.1. Иконка уведомлений

- Колокольчик в шапке (top-right).
- Бейдж с количеством непрочитанных (>0).
- При клике — dropdown с последними 10 уведомлениями.
- Каждое уведомление: иконка по типу, заголовок, текст, время («5 мин назад», «2 ч назад», «вчера»).
- Кнопки: «Отметить все прочитанными», «Все уведомления».

### 13.4.2. Страница «Все уведомления»

- Таблица: Тип | Заголовок | Текст | Дата | Статус (прочитано/нет).
- Фильтры: тип, период, статус.
- Массовые действия: «Отметить прочитанными», «Удалить».

### 13.4.3. Страница «Согласования»

- Две вкладки: «Заявки от меня» / «Заявки на меня».
- Таблица: Дата | Тип | Сущность | Сумма | Инициатор | Статус | Действия.
- В/actions: «Одобрить», «Отклонить» (с модалом комментария).
- Фильтры: статус, тип, период.

### 13.4.4. Настройки уведомлений

- Таблица: Тип уведомления | In-app | Email | Telegram (чекбоксы).
- Сохранение настроек по пользователю.

## 13.5. API-эндпоинты

```
GET    /api/v1/notifications                 — список (с фильтрами)
POST   /api/v1/notifications/:id/read        — отметить прочитанным
POST   /api/v1/notifications/read-all        — отметить все прочитанными
DELETE /api/v1/notifications/:id             — удаление
GET    /api/v1/notifications/unread-count    — количество непрочитанных

GET    /api/v1/notification-settings         — настройки текущего пользователя
PUT    /api/v1/notification-settings         — обновление

GET    /api/v1/approvals                     — список заявок { direction: 'mine' | 'to_me' }
POST   /api/v1/approvals                     — создание заявки
GET    /api/v1/approvals/:id                 — карточка
POST   /api/v1/approvals/:id/approve         — одобрение
POST   /api/v1/approvals/:id/reject          — отклонение { comment }
POST   /api/v1/approvals/:id/cancel          — отзыв инициатором

POST   /api/v1/telegram/link                 — привязка chatId { code }
GET    /api/v1/telegram/status               — статус привязки
```

## 13.6. Технические аспекты

### 13.6.1. Архитектура

```
Событие в системе (например, сделка перешла на новую стадию)
  ↓
Next.js создаёт запись в Notification
  ↓
Next.js отправляет webhook в Python-воркер: POST /internal/notify { notificationId }
  ↓
Python-воркер проверяет настройки пользователя
  ↓
Отправляет in-app (через WebSocket или polling) + email (aiosmtplib) + Telegram (python-telegram-bot)
  ↓
Логирует результаты отправки в SyncLog
```

### 13.6.2. WebSocket (опционально)

Для real-time обновлений — отдельный endpoint `/ws/notifications`:
- При подключении — авторизация по JWT.
- Сервер пушит новые уведомления для этого пользователя.
- Fallback: polling каждые 30 секунд на `/api/v1/notifications/unread-count`.

### 13.6.3. Шаблоны сообщений

Шаблоны email и Telegram-сообщений хранятся в `Setting` как JSON:

```json
{
  "deal_stage_changed": {
    "email_subject": "Сделка {{dealNumber}} перешла на стадию {{stageName}}",
    "email_body": "Здравствуйте, {{userName}}! Сделка {{dealTitle}} ({{dealNumber}}) перешла на стадию {{stageName}}. Сумма: {{amount}}. Ответственный: {{managerName}}.",
    "telegram_text": "Сделка *{{dealNumber}}* перешла на стадию *{{stageName}}*\nСумма: {{amount}}\nМенеджер: {{managerName}}"
  }
}
```

## 13.8. Метрики модуля

- Количество отправленных уведомлений по типам и каналам.
- Доля прочитанных в течение 1 часа.
- Среднее время согласования заявки (от создания до решения).
- Доля отклонённых заявок.
- Доставляемость email (без ошибок) — >98%.
