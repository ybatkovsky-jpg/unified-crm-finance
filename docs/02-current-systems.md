# 02. Анализ существующих систем

Этот раздел фиксирует текущее состояние программных комплексов zakuppro и finpro: их стек, сущности, функциональность, интеграционные точки и недостатки. Анализ служит основанием для проектных решений в последующих разделах.

## 2.1. zakuppro — система управления закупками

### 2.1.1. Назначение

zakuppro автоматизирует полный цикл от спецификации до оплаты в строительных и монтажных проектах: загрузка спецификации (BOM) через Telegram-бот или веб-интерфейс, AI-распознавание структуры, группировка позиций по поставщикам, автоматическая отправка email-запросов, обработка входящих счетов через IMAP, сверка счетов с платежами по банковским выпискам 1С-клиент-банк, резервирование и списание со склада, Kanban-доска проектов с guardrails.

### 2.1.4. Архитектура

zakuppro построен как **модульный монолит** с явным разделением на 5 контуров:

1. **Web App** (Next.js + FastAPI backend) — интерфейс для менеджеров: Kanban-доски, таблицы спецификаций, ручная сверка счетов, склад.
2. **Telegram Bot Gateway** — приём Excel-файлов от владельца, отправка счетов на оплату, пуш-алерты.
3. **AI-Agent Worker** (Celery) — парсинг Excel, формирование промптов LLM, вызов API, принятие решений, сохранение результатов.
4. **Email Worker** (IMAP/SMTP) — отправка запросов поставщикам, опрос ящика каждые N минут, маршрутизация входящих счетов в очередь на сверку.
5. **Bank Worker** (Cron) — раз в сутки забирает выписку (1С-клиент-банк), мапит платежи по ИНН и сумме.

Backend содержит 17 роутеров, 14 бизнес-сервисов, 16 ORM-моделей, Alembic-миграции, 34 тестовых файла. Имеется DLQ для failed Celery-задач и админка для их перезапуска.

### 2.1.5. Функциональность

- Загрузка BOM через Telegram-бота и веб-интерфейс.
- AI-парсинг Excel (Pandas для чистых таблиц, LLM для грязных), нормализация названий, определение артикулов и поставщиков.
- Группировка позиций по поставщикам, формирование запросов в один клик.
- Email-рассылка запросов с BCC на рабочую почту, логирование переписки в `EmailLog`.
- Обработка входящих писем с вложениями (PDF/Excel), извлечение табличной части счёта через LLM, fuzzy-matching с заказом (Levenshtein + embeddings).
- Импорт банковской выписки (1С-клиент-банк), автоматическое связывание платежей со счетами по ИНН+сумма, ручная привязка нераспознанных.
- Склад с резервированием: при создании BOM система списывает `qty_available` → `qty_reserved`; при выдаче в производство — `qty_reserved` → 0, `qty_total` уменьшается.
- Kanban-доска проектов с guardrails: переход в «Производство» блокируется, если есть незакрытые позиции (если не стоит галка «Игнорировать нехватку» от владельца).
- Матрица готовности проекта с цветовой индикацией (зелёный/жёлтый/красный).
- RBAC: владелец / менеджер / склад.
- Telegram-уведомления о статусах (счёт получен, оплачен, поставка прибыла).

### 2.1.6. Недостатки и ограничения

1. **Нет понятия «клиент» и «сделка»** — проект создаётся сразу под клиента без воронки. Все лиды попадают «сверху», без фиксации первого контакта.
2. **Нет договоров** — только `customerName` строкой в `Project`, без генерации документов и версионности.
3. **Финансовый учёт ограничен счетами и платежами** — нет бюджетов, нет кассовых разрывов, нет P&L.
4. **Дублирование `Supplier`/`Counterparty`** — в finpro есть отдельная таблица `Counterparty`, в zakuppro — `Supplier`. При объединении потребуется консолидация.
5. **Слабая интеграция с finpro** — только pull-синхронизация проектов из finpro в одну сторону; нет обратной связи по фактическим затратам.
6. **Проблема нескольких юрлиц** — `CompanyDetails` поддерживает несколько, но финансовый учёт ведётся без разделения по юрлицам.
7. **Слабый аудит** — нет единой таблицы аудита всех действий (история статусов проекта есть, но не покрыты изменения счетов, склада, настроек).
8. **AI-промпты захардкожены** — нет UI для редактирования системных промптов и тестирования их на реальных данных.

## 2.2. finpro — финансовый контроль с элементами CRM

### 2.2.1. Назначение

finpro ведёт финансовый управленческий учёт по проектам: импорт транзакций из 1С-клиент-банк, ручная классификация, бюджеты по проектам и категориям, кассовые разрывы (план/факт по датам), P&L по проекту и по бизнесу в целом, аудит всех действий, RBAC с ролями owner/accountant/manager/storekeeper. Содержит базовый модуль клиентов (`Client`) и синхронизацию проектов из zakuppro.

### 2.2.2. Стек технологий

|------|------------|--------|
| Frontend | Next.js (App Router) | 16.1 |

| UI framework | Tailwind CSS + shadcn/ui (40+ Radix-компонентов) | 4 |
| State management | Zustand | 5 |
| Data fetching | @tanstack/react-query | 5 |
| Tables | @tanstack/react-table | 8 |
| Charts | Recharts | 2 |
| DnD | @dnd-kit (в Kanban-представлениях) | 6 |

### 2.2.3. Модель данных (14 сущностей)

| Сущность | Назначение |
|----------|------------|
| `User` | Пользователь: email, имя, пароль (bcrypt), роль (owner/accountant/manager/storekeeper), isActive |

| `Transaction` | Транзакция: проект, категория, контрагент, дата, сумма, тип (income/expense), источник (manual/1c_clientbank/zakuppro), внешний ID для дедупликации |

| `CashFlowPayment` | Плановый/фактический платёж: дата, сумма, тип (inflow/outflow/planned_*), контрагент, проект, статус (planned/confirmed/cancelled) |
| `Notification` | Уведомление пользователю: тип, заголовок, текст, isRead, ссылка |
| `SyncLog` | Журнал синхронизаций: источник (zakuppro/1c_clientbank/csv), статус, кол-во записей, ошибки |

### 2.2.4. Архитектура

finpro построен как **Next.js fullstack-приложение**: вся бизнес-логика в API Routes (`src/app/api/`), доступ к данным через Prisma. Аутентификация — NextAuth с Credentials-провайдером и JWT-сессиями. RBAC реализован через middleware + helper-функции `requireRole(user, 'owner')`. Имеется rate limiting (in-memory, по IP и userId), CSRF-защита, sanitization ввода, audit log на ключевые действия. Reverse proxy — Caddy с автоматическим HTTPS. Production-сборка запускается на Bun runtime для производительности.

### 2.2.5. Функциональность

- Авторизация по email/паролю, refresh-токены, ролевая модель (4 роли).
- CRUD клиентов, контрагентов, категорий (иерархия), проектов.
- CRUD транзакций с привязкой к проекту/категории/контрагенту, ручной ввод.
- Импорт 1С-клиент-банк (XML), парсинг в `Transaction` с дедупликацией по `externalId`.
- Импорт CSV (общий формат), маппинг колонок.
- Классификация: правила по ключевым словам, авто-привязка к проекту и категории.
- Бюджеты: план/факт по проекту × категории × периоду.
- Кассовые разрывы: плановые и фактические платежи по датам, индикация разрывов.
- P&L по проекту: доходы минус расходы с фильтром по периоду, экспорт в Excel.
- P&L по бизнесу: агрегация по всем проектам, экспорт в Excel.
- Дашборд: ключевые метрики (выручка, расходы, маржа, кассовый разрыв), графики.
- Уведомления: бюджетный перерасход, дедлайны проектов, ошибки синхронизации, кассовые разрывы.
- Аудит: журнал всех изменений с возможностью отката (read-only).
- Поиск по системе (command palette).
- Управление пользователями (только owner).
- Закрытие периодов с защитой от изменений.
- API-key аутентификация для интеграций.
- Синхронизация с zakuppro: pull проектов с маппингом статусов.

### 2.2.6. Недостатки и ограничения

1. **Нет полноценного CRM** — `Client` хранит только имя/ИНН/телефон/email; нет воронки, стадий, ответственного менеджера, истории взаимодействий.
2. **Нет договоров** — только поле `contractAmount` в `Project`; нет шаблонов, генерации PDF, версионности.
3. **Нет закупок** — нет понятия спецификации (BOM), поставщиков, запросов, счетов от поставщиков. Только готовые транзакции.
4. **Нет склада** — нет остатков, резервирования, списания.
5. **Нет AI/LLM-агента** — вся классификация ручная или по жёстким правилам.
6. **Нет Telegram-бота** — уведомления только in-app.
7. **SQLite в dev** — некоторые SQL-конструкции (например, `JSONB`) недоступны, приходится дублировать схемы (`schema.sqlite.prisma` и `schema.postgresql.prisma`).
8. **Слабая синхронизация с zakuppro** — односторонняя, нет push-обновлений, нет связывания транзакций со счетами zakuppro.
9. **Нет согласований** — нет workflow «заявка на закупку → согласование → оплата».

## 2.4. Выводы для проектирования единой системы

1. **Frontend-стек идентичен** — это ключевое преимущество: одна кодовая база UI без переписывания. Нужно лишь унифицировать дизайн-токены и навигацию.
2. **Backend-стеки разные, но комплементарные** — Next.js API Routes удобен для CRUD и UI-связанной логики; Python FastAPI — для AI, email, парсинга Excel/PDF, Celery-задач. Решение: гибридная архитектура, где Next.js — основной API, Python — отдельный «воркерный» контур.
3. **Модели данных нужно сливать** — `Supplier` (zakuppro) и `Counterparty` (finpro) → единый `Counterparty` с типом; `Project` обеих систем → единый `Project` с расширенными полями; `Client` (finpro) → расширить до `Contact` (CRM) с типом (client/lead/person).
4. **Интеграцию нужно делать двусторонней** — после подписания договора в CRM автоматически создаётся проект в закупках; после проведения платежа в финансах обновляется статус счёта в закупках.
5. **AI-агента и email-воркер переносим как есть** — это ценная функциональность, не нужно переписывать на TS. Оборачиваем новые таблицы единой БД.
6. **Аутентификация — на основе finpro** — NextAuth + JWT + RBAC уже работает; добавляем роль `sales` для менеджеров по продажам.
7. **Склад и Kanban guardrails — из zakuppro** — эта логика отлажена и проверена; переносим с минимальными изменениями.
8. **Бюджеты, P&L, кассовые разрывы — из finpro** — финансовый контур целиком берём оттуда.
9. **CRM, Сделки, Договоры — разработка с нуля** — этих модулей нет ни в одной из исходных систем.
10. **Миграция данных — старт с чистого листа** (по решению заказчика). Старые системы остаются в архивном режиме на 2–4 недели, затем выводятся из эксплуатации.
