# 07. Модуль Сделки и воронка

## 7.1. Назначение

Модуль Сделки — канбан-воронка, в которой ведётся работа с потенциальными заказами от первого контакта до подписания договора или отказа. Это «кровеносная система» продаж: позволяет менеджерам видеть свою загрузку, прогнозировать выручку, идентифицировать «зависшие» сделки.

Каждая сделка связана с контактом (`Contact`), может привести к договору (`Contract`) и проекту (`Project`). Сделка движется по стадиям воронки (`Pipeline` → `DealStage`), при переходе на стадию «Выиграно» автоматически создаётся договор (опционально).

## 7.2. Функциональные требования

### 7.2.1. Воронки и стадии

| ID | Требование | Приоритет |
|----|------------|-----------|
| DEAL-01 | Система должна поддерживать несколько воронок (`Pipeline`): например, «B2B-продажи», «Розница», «Сервис» | Should |
| DEAL-02 | Каждая воронка должна иметь упорядоченный набор стадий (`DealStage`) с вероятностью выигрыша в % | Must |
| DEAL-03 | Стадии должны иметь признак «выигрышная» (`isWonStage`) и «проигрышная» (`isLostStage`) — для расчёта конверсии | Must |
| DEAL-04 | Должна быть предустановленная воронка «По умолчанию» со стадиями: Новый лид → Квалификация → Встреча → КП → Переговоры → Договор → Выиграно / Потеряно | Must |
| DEAL-05 | Администратор должен иметь возможность редактировать стадии (имя, вероятность, цвет, порядок) | Should |
| DEAL-06 | Система должна блокировать удаление стадии, на которой есть сделки (предлагать слить с другой) | Must |

### 7.2.2. Управление сделками

| ID | Требование | Приоритет |
|----|------------|-----------|
| DEAL-07 | Система должна позволять создавать сделку с полями: заголовок*, контакт*, воронка*, стадия*, сумма*, ожидаемая дата закрытия, ответственный менеджер, описание | Must |
| DEAL-08 | Система должна автоматически генерировать номер сделки в формате `С-2026-00001` (нумератор по году) | Must |
| DEAL-09 | Система должна позволять перемещать сделку между стадиями перетаскиванием (drag-and-drop) на Kanban-доске | Must |
| DEAL-10 | При перемещении сделки на выигрышную/проигрышную стадию — требовать причину (для проигрыша) и фиксировать `actualCloseDate` | Must |
| DEAL-11 | Система должна вести историю перемещений сделки по стадиям (`DealHistory`): когда, кем, с комментарием | Must |
| DEAL-12 | Система должна позволять назначать нескольких наблюдателей (watchers) за сделкой (помимо ответственного) | Should |
| DEAL-13 | Система должна поддерживать soft-delete сделок | Must |
| DEAL-14 | Система должна автоматически переводить сделку в статус «Выиграно» при создании связанного договора | Must |
| DEAL-15 | Система должна показывать на карточке сделки: прогресс по стадиям (визуально), контакты, взаимодействии, договор (если есть), проект (если есть) | Must |

### 7.2.3. Kanban-доска

| ID | Требование | Приоритет |
|----|------------|-----------|
| DEAL-16 | Система должна отображать сделки в виде Kanban-доски: колонки = стадии, карточки = сделки | Must |
| DEAL-17 | Карточка сделки должна показывать: заголовок, сумма, контакт, ответственный (аватар), дата создания, иконки («есть новая активность», «прикреплён договор») | Must |
| DEAL-18 | Система должна позволять фильтровать Kanban: по ответственному, по контакту, по сумме (диапазон), по ожидаемой дате закрытия | Must |
| DEAL-19 | Система должна позволять переключаться между воронками (если их несколько) | Must |
| DEAL-20 | Система должна показывать над каждой колонкой сумму сделок в ней и количество | Must |
| DEAL-21 | Перетаскивание сделки на выигрышную/проигрышную колонку должно вызывать модал подтверждения с причиной | Must |
| DEAL-22 | Система должна поддерживать перетаскивание только вперёд или назад по стадиям (не через одну) — настраивается в админке | Should |
| DEAL-23 | Система должна позволять переключаться в табличный вид (для массовых операций) | Must |

### 7.2.4. Прогноз и аналитика

| ID | Требование | Приоритет |
|----|------------|-----------|
| DEAL-24 | Система должна считать прогноз выручки: сумма сделки × вероятность стадии, агрегированно по менеджеру/воронке/периоду | Must |
| DEAL-25 | Система должна показывать конверсию воронки: % перехода между стадиями, общая конверсия лид→договор | Must |
| DEAL-26 | Система должна показывать средний цикл сделки (в днях) от создания до закрытия | Should |
| DEAL-27 | Система должна идентифицировать «зависшие» сделки (без смены стадии >14 дней) и подсвечивать их | Should |

## 7.3. User Stories

- **US-DEAL-01**: Как менеджер по продажам, я хочу видеть все свои сделки на Kanban-доске, чтобы понимать, на каких стадиях сколько денег «висит».
- **US-DEAL-02**: Как менеджер, я хочу перетащить сделку на стадию «Договор», чтобы автоматически создать договорный документ.
- **US-DEAL-03**: Как владелец, я хочу видеть прогноз выручки на следующий квартал на основе вероятностей сделок.
- **US-DEAL-04**: Как менеджер, я хочу получить уведомление о сделке, которая «зависла» на 2 недели, чтобы связаться с клиентом.
- **US-DEAL-05**: Как владелец, я хочу видеть конверсию по каждому менеджеру, чтобы оценить эффективность работы.

## 7.4. UI/UX

### 7.4.1. Kanban-доска

Шапка:
- Selector воронки (если несколько).
- Фильтры: ответственный, контакт, сумма, дата закрытия.
- Кнопка «Вид: Kanban / Таблица».
- Кнопка «Создать сделку».

Доска (горизонтальный скролл):
- Колонки = стадии, ширина фиксированная (~280px).
- В шапке колонки: имя стадии (с цветным бейджем), сумма сделок, количество.
- Карточки внутри колонки, вертикально, скролл внутри колонки.
- Карточка: заголовок, сумма (крупно), контакт (с аватаром), ответственный аватар внизу справа, бейджи («новая активность», «привязан договор»).
- DnD между колонками.

### 7.4.2. Карточка сделки (drawer справа)

- Заголовок крупно, номер.
- Сумма, ожидаемая дата закрытия.
- Контакт (кликабельная мини-карточка).
- Ответственный, наблюдатели.
- Прогресс по стадиям (горизонтальный stepper).
- Описание.
- Табы:
  - Активности (взаимодействия).
  - Договор (если есть, или кнопка «Создать договор»).
  - Проект (если есть).
  - История (перемещения по стадиям).
  - Файлы.

### 7.4.3. Модал создания сделки

- Заголовок* (input).
- Контакт* (select с поиском, кнопка «Создать новый»).
- Воронка* (select).
- Стадия* (select, зависит от воронки).
- Сумма* (number + валюта).
- Ожидаемая дата закрытия (date picker).
- Ответственный* (select).
- Описание (textarea).

## 7.7. Бизнес-правила

### 7.7.1. Автоматические действия

1. При переходе сделки на стадию «Договор» с `isWonStage=false` (т.е. это ещё не финальная стадия) — предложить создать договор.
2. При переходе на стадию с `isWonStage=true` — требовать `actualCloseDate` (по умолчанию сегодня), опционально создать договор.
3. При переходе на стадию с `isLostStage=true` — требовать `lossReason` (select: «дорого», «выбрали конкурента», «отказ», «нецелевой лид», «другое»).
4. При создании договора из сделки — `Deal.contractId` проставляется автоматически, статус сделки → финальная выигрышная.

### 7.7.2. Валидации

- `amount >= 0` (может быть 0 для оценочных).
- `expectedCloseDate >= createdAt`.
- `contactId` обязателен (нельзя создать сделку без контакта).
- `stageId` должен принадлежать `pipelineId` сделки.

### 7.7.3. Сквозные связи

- Сделка → Договор (1:1, через `contractId`).
- Сделка → Проект (1:1, через `projectId` — после создания проекта из сделки).
- Сделка → Контакт (N:1).
- Сделка → Взаимодействия (через `Contact`).

## 7.8. Метрики модуля

- Количество активных сделок (по стадиям, менеджерам, воронкам).
- Сумма сделок в работе (weighted forecast = сумма × вероятность).
- Конверсия по стадиям (из скольких сделок на стадии N перешли на N+1).
- Средний цикл сделки (дни от создания до закрытия).
- Динамика создания сделок по неделям/месяцам.
- Win rate = выигранные / (выигранные + проигранные) за период.
- ТОП-5 «зависших» сделок (по давности последней активности).
