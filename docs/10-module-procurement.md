# 10. Модуль Закупки

## 10.1. Назначение

Модуль Закупки — самый функционально насыщенный модуль, переносимый из zakuppro почти без изменений. Он покрывает полный цикл: спецификация (BOM) → запрос поставщику → счёт → сверка → оплата → поставка → склад. Включает AI-агента для парсинга Excel и сверки счетов, email-воркер для отправки запросов и приёма счетов, склад с резервированием.

В unified-системе модуль интегрируется с CRM (через проект и контрагента), с договорами (сумма договора = бюджет проекта), с финансами (счета → транзакции, остатки на складе → оценка активов).

## 10.3. User Stories

- **US-PROC-01**: Как менеджер, я хочу загрузить Excel-спецификацию в проект, чтобы AI-агент распознал позиции и я мог отправить запросы поставщикам.
- **US-PROC-02**: Как закупщик, я хочу получить счёт от поставщика на email и увидеть его в системе уже сверённым, чтобы не тратить время на ручной ввод.
- **US-PROC-03**: Как владелец, я хочу видеть заявку на оплату счёта >100k, чтобы лично подтвердить.
- **US-PROC-04**: Как кладовщик, я хочу видеть приход от поставщика и подтвердить приёмку, чтобы обновить остатки.
- **US-PROC-05**: Как менеджер, я хочу получить уведомление, что позиция из спецификации уже есть на складе, чтобы не заказывать её повторно.

## 10.7. AI-агент (детали реализации)

### 10.7.1. Парсинг Excel

1. Файл сохраняется в S3, создаётся `FileEntity`.
2. Next.js отправляет webhook в Python-воркер: `POST /internal/ai/parse-bom { fileId, projectId }`.
3. Python-воркер скачивает файл, открывает через `pandas.read_excel()`.
4. Если структура чистая (есть заголовки) — Pandas напрямую создаёт DataFrame.
5. Если структура грязная (объединённые ячейки, многоэтажные шапки) — LLM-промпт: «Распознай структуру таблицы и верни JSON».
6. Нормализация названий через LLM (убрать мусор, выделить артикул).
7. Для каждой позиции — попытка найти поставщика в справочнике по категории.
8. Сохранение `BOMItem` в БД.
9. Webhook обратно в Next.js: `POST /internal/notifications { userId, type: 'bom_parsed', projectId, itemsCount }`.

### 10.7.2. Сверка счетов

1. Email-воркер сохраняет вложение в S3, создаёт `Invoice` со статусом `received`.
2. Отправляется webhook в AI-воркер: `POST /internal/ai/verify-invoice { invoiceId }`.
3. PDF/Excel парсится (`pdfplumber` для PDF, `pandas` для Excel).
4. LLM извлекает позиции: «Верни JSON-массив с { name, quantity, price, article? }».
5. Для каждой позиции счёта — fuzzy-matching с `PurchaseRequestItem` через `rapidfuzz` (Levenshtein) и embeddings.
6. Если совпадение >90% — `isMatch=true`. Если 70-90% — флаг `mismatchReason`. Если <70% — помечается как новая позиция.
7. Если все позиции совпали — статус `verified`. Иначе `discrepancy` с diff.
8. Webhook в Next.js: `POST /internal/notifications { userId, type: 'invoice_verified', invoiceId, status }`.

### 10.7.3. LLM-провайдеры (multi-provider)

- DeepSeek Chat — основной (дешёвый, хорошо работает со структурированным текстом).
- OpenAI GPT-4o — fallback 1.
- Anthropic Claude 3.5 — fallback 2.
- Google Gemini — fallback 3.
- Qwen Plus — fallback 4.
- Переключение через `@retry_async` с экспоненциальной задержкой.
- Все вызовы логируются в `LlmLogs` (отдельная таблица для аудита расходов).

## 10.8. Email-воркер (детали)

- Опрос IMAP каждые 15 минут (настраивается в `Setting`).
- Фильтр: только непрочитанные письма от известных поставщиков (по домену email) или с темой, содержащей «счёт» / «счет» / «invoice».
- Вложения сохраняются в S3, создаётся `FileEntity`.
- Создаётся `Invoice` со статусом `received`, `sourceFileId`.
- Запускается AI-сверка.
- Письмо помечается прочитанным, пишется в `EmailLog`.

## 10.9. Метрики модуля

- Количество BOM в обработке / обработано.
- Среднее время от загрузки Excel до отправки запросов (минуты).
- Доля счетов с автоматической сверкой (без ручной правки).
- Средний срок ответа поставщика (дни от запроса до счёта).
- Сумма заявок на оплату в работе / одобрено / отклонено.
- Оборачиваемость склада (по категориям).
- Сумма кассовых разрывов из-за несвоевременной оплаты счетов.
