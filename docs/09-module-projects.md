# 09. Модуль Проекты/объекты

## 9.1. Назначение

Модуль Проекты — основной операционный контур после подписания договора. Проект объединяет всё: спецификацию (BOM), закупки, склад, производство/монтаж, финансовый учёт. Это «зонтик» под которым собирается вся работа по конкретному объекту.

Проект создаётся из договора (или напрямую) и проходит жизненный цикл от стадии «Проектирование» до «Сдача». Kanban-доска проектов с guardrails (как в zakuppro) не даёт пропустить этапы (например, нельзя начать производство, если есть незакрытые закупки).

## 9.2. Функциональные требования

### 9.2.1. Создание и жизненный цикл

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-01 | Система должна позволять создавать проект: из договора (кнопка «Создать проект»), или напрямую | Must |
| PROJ-02 | При создании из договора — автоподстановка: контакт, контрактная сумма, дата начала | Must |
| PROJ-03 | Система должна автоматически генерировать внешний номер проекта в формате `ПМ-2026-00001` | Must |
| PROJ-04 | Поля проекта: номер, имя, описание, сделка, договор, контакт, статус, контрактная сумма, целевая маржа, дата начала, дата окончания, дата завершения, оценка качества, статус дедлайна, ответственный менеджер | Must |
| PROJ-05 | Статусы: `lead` → `active` → `on_hold` → `completed` / `cancelled` | Must |
| PROJ-06 | Kanban-доска проектов с guardrails: нельзя перейти на следующую стадию, если есть блокеры | Must |
| PROJ-07 | Система должна требовать причину при переходе в `on_hold` или `cancelled` | Must |
| PROJ-08 | При переходе в `completed` — требовать `qualityRating` (good/acceptable/poor) и `completedAt` | Must |
| PROJ-09 | Система должна автоматически считать `deadlineStatus`: `on_track` (до дедлайна >30% времени), `at_risk` (<30%), `overdue` (просрочен) | Must |

### 9.2.2. Этапы проекта

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-10 | Каждый проект должен иметь предустановленный набор этапов (`ProjectStage`): Проектирование → Закупки → Производство → Монтаж → Сдача | Must |
| PROJ-11 | Этапы можно настраивать (добавлять/удалять/переупорядочивать) на уровне проекта | Should |
| PROJ-12 | Каждый этап имеет: статус (pending/in_progress/completed/skipped), дату начала, дату окончания, ответственного, заметки | Must |
| PROJ-13 | Система должна считать прогресс проекта: % завершённых этапов (по количеству или по весам) | Must |
| PROJ-14 | Система должна автоматически переводить этап в `in_progress` при первой операции в нём (например, создание первой закупки → этап «Закупки» активен) | Should |

### 9.2.3. Команда проекта

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-15 | Система должна позволять назначать участников команды проекта (`ProjectMember`): менеджер, закупщик, бухгалтер, кладовщик, исполнитель | Must |
| PROJ-16 | Один пользователь может быть в нескольких ролях в одном проекте | Must |
| PROJ-17 | Участники видят проект в своём списке «Мои проекты» | Must |
| PROJ-18 | Система должна позволять исключать участников (`leftAt`) с сохранением истории | Must |
| PROJ-19 | Только owner и менеджер проекта могут редактировать состав команды | Must |

### 9.2.4. Финансовая сводка

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-20 | В карточке проекта должна отображаться финансовая сводка: контрактная сумма, плановые затраты (бюджет), фактические затраты, фактические поступления, текущая маржа, отклонение от плана | Must |
| PROJ-21 | Система должна считать маржу: `(contractAmount - totalCost) / contractAmount × 100%` | Must |
| PROJ-22 | Система должна подсвечивать красным, если маржа ниже целевой (`marginTarget`) | Must |
| PROJ-23 | Система должна показывать кассовый поток по проекту: плановые и фактические поступления/платежи по датам | Must |
| PROJ-24 | Система должна показывать P&L проекта с разбивкой по категориям затрат | Must |

### 9.2.5. Kanban guardrails (бизнес-правила)

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-25 | Переход из «Закупки» в «Производство» блокируется, если есть `BOMItem` со статусом ниже `delivered` (или `available` если со склада) | Must |
| PROJ-26 | Опционально: владелец может обойти guardrail через галку «Игнорировать нехватку» (с обязательным комментарием, фиксируется в `AuditLog`) | Must |
| PROJ-27 | Переход в «Сдача» блокируется, если есть незакрытые счета (`Invoice.status != paid` или `cancelled`) | Must |
| PROJ-28 | Система должна показывать модальное окно с конкретным списком блокеров при попытке перехода | Must |

### 9.2.6. Матрица готовности

| ID | Требование | Приоритет |
|----|------------|-----------|
| PROJ-29 | Система должна показывать экран «Матрица готовности» для проекта: таблица `BOMItem` × статус с цветовой индикацией (зелёный = доставлено/в наличии, жёлтый = в пути/заказано, красный = не заказано/ожидает) | Must |
| PROJ-30 | Система должна позволять фильтровать матрицу по поставщику, категории, статусу | Must |
| PROJ-31 | Система должна показывать общий процент готовности проекта (по позициям) | Must |

## 9.3. User Stories

- **US-PROJ-01**: Как менеджер, я хочу видеть все свои активные проекты на Kanban-доске, чтобы понимать, какие на каком этапе.
- **US-PROJ-02**: Как менеджер, я хочу увидеть модальное окно с причиной блокировки перехода проекта на следующий этап, чтобы знать, что нужно доделать.
- **US-PROJ-03**: Как владелец, я хочу видеть маржу по каждому проекту в реальном времени, чтобы вовремя заметить убыточные.
- **US-PROJ-04**: Как закупщик, я хочу видеть «Матрицу готовности», чтобы понимать, какие позиции ещё нужно закупить.
- **US-PROJ-05**: Как менеджер, я хочу получить уведомление о приближении дедлайна проекта, чтобы успеть сдать вовремя.

## 9.4. UI/UX

### 9.4.1. Kanban-доска проектов

Аналогично сделкам (см. [docs/07](07-module-deals.md)), но:
- Колонки = этапы (Проектирование → Закупки → Производство → Монтаж → Сдача → Завершён).
- Карточка проекта: номер, имя, клиент, контрактная сумма, маржа (с цветом), аватар менеджера, бейдж дедлайна (зелёный/жёлтый/красный).
- При попытке DnD на следующую колонку с блокером — модальное окно.

### 9.4.2. Карточка проекта

Шапка:
- Номер, имя, статус (бейдж).
- Прогресс-бар этапов.
- Кнопки: Редактировать, Архивировать.

Сводка (верхняя часть):
- 4 карточки-метрики: Контрактная сумма, Плановые затраты, Фактические затраты, Маржа.
- Под метриками — мини-график кассового потока.

Табы:
1. **Обзор** — основные поля, этапы (stepper), команда.
2. **Спецификация (BOM)** — таблица позиций, кнопка «Загрузить Excel».
3. **Закупки** — список запросов поставщикам и счетов.
4. **Склад** — движения по проекту.
5. **Финансы** — транзакции, бюджеты, P&L.
6. **Договор** — связанный договор.
7. **Команда** — участники с ролями.
8. **История** — audit log.

### 9.4.3. Матрица готовности

Отдельный экран (доступен из карточки проекта):
- Таблица: Позиция | Артикул | Категория | Количество | Поставщик | Статус | Цвет.
- Фильтры по поставщику, категории, статусу.
- Снизу — общая статистика: готовность X% (Y из Z позиций).

## 9.5. API-эндпоинты

```
GET    /api/v1/projects                    — список (с фильтрами)
POST   /api/v1/projects                    — создание
GET    /api/v1/projects/:id                — карточка
PUT    /api/v1/projects/:id                — обновление
DELETE /api/v1/projects/:id                — soft delete
POST   /api/v1/projects/:id/move           — смена этапа { toStageCode, comment, ignoreBlockers }
GET    /api/v1/projects/:id/stages         — этапы
PUT    /api/v1/projects/:id/stages/:sid    — редактирование этапа
GET    /api/v1/projects/:id/members        — команда
POST   /api/v1/projects/:id/members        — добавление участника
DELETE /api/v1/projects/:id/members/:uid   — удаление участника
GET    /api/v1/projects/:id/readiness-matrix — матрица готовности
GET    /api/v1/projects/:id/financial-summary — финансовая сводка
GET    /api/v1/projects/:id/cashflow       — кассовый поток
GET    /api/v1/projects/:id/pnl            — P&L
```

## 9.6. Роли и права

| Действие | owner | sales | manager | accountant | storekeeper |

| Просмотр проектов | все | свои (если в команде) | все | все | свои (если в команде) |
| Создание | да | да | да | нет | нет |
| Редактирование полей | да | нет | свои | нет | нет |
| Смена этапа | да | нет | свои | нет | нет |
| Управление командой | да | нет | свои | нет | нет |
| Загрузка BOM | да | нет | да | нет | нет |
| Просмотр финансов | да | нет | да | да | нет |
| Удаление | да | нет | нет | нет | нет |

## 9.7. Связи с другими модулями

- **CRM**: `Project.contactId` → `Contact`.
- **Sales**: `Project.dealId` → `Deal` (опционально).
- **Contracts**: `Project.contractId` → `Contract` (опционально).
- **Procurement**: `BOM.projectId` → `Project` (1:1), `PurchaseRequest.projectId`, `Invoice.projectId`, `Delivery.projectId`.
- **Finance**: `Transaction.projectId`, `Budget.projectId`, `CashFlowPayment.projectId`.

## 9.8. Метрики модуля

- Количество активных проектов по этапам.
- Сумма контрактов по активным проектам.
- Средняя маржинальность по завершённым проектам.
- Доля проектов с просрочкой дедлайна.
- ТОП-5 проектов с низкой маржой (< целевой).
- Длительность каждого этапа (среднее).
